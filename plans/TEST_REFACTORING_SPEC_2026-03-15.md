# Спецификация рефакторинга тестового набора

Дата: 2026-03-15
Статус: Draft for approval

## 1. Цель

Перестроить тестовый набор так, чтобы он:

1. защищал реальные пользовательские сценарии, а не только внутренние функции;
2. покрывал критические failure-path и success-path в DOCX-конвейере;
3. убирал устаревшие, мёртвые и избыточно хрупкие тесты;
4. снижал drift между кодом, тестами и контрактной документацией;
5. оставался быстрым, предсказуемым и пригодным для CI.

## 2. Основание для рефакторинга

Аудит и последующая валидация замечаний подтвердили, что текущий набор тестов силён в областях DOCX semantics и image-pipeline routing, но имеет системный перекос:

- сильное покрытие внутренних компонентов;
- слабое покрытие сквозного пользовательского success-path;
- наличие stale-тестов на уже несуществующий UI-контракт;
- наличие implementation-detail тестов, фиксирующих частную реализацию вместо внешнего поведения;
- частичное расхождение между документацией и фактическим contract-suite.

## 3. Подтверждённые выводы аудита

### 3.1 Сильные стороны текущего набора

Наиболее полезная и актуальная часть регрессий сосредоточена вокруг:

- `document.py` — извлечение DOCX, placeholder-контракт, reinsertion изображений, структурная устойчивость;
- `document_pipeline.py` — контроль опасных failure-mode в основном пайплайне;
- `preparation.py` и `processing_runtime.py` — подготовка, кеширование, single-flight, фоновые переходы состояний;
- image-suite — policy routing, fallback, validation, compare-all, интеграция image-stage.

### 3.2 Подтверждённые проблемы

Подтверждены следующие системные недостатки:

1. Отсутствует один реальный сквозной integration-test на сценарий upload → processing → output DOCX.
2. Тесты compare-panel закрепляют старый контракт, которого больше нет.
3. Функция `render_image_compare_selector` является dead code: используется только в `ui.py` и собственном тесте.
4. Ряд UI-тестов зависит от дословных CSS-строк и потому хрупок.
5. `test_app_runtime.py` проверяет тонкий wiring и имеет низкую продуктовую ценность.
6. `test_processing_service.py` смещён в сторону crash-path и почти не страхует успешное завершение worker-потока.
7. spec-like тесты проверяют в основном наличие текстовых артефактов и плохо коррелируют с поведением приложения.
8. В `test_preparation.py` есть как минимум один тест, который фиксирует детали реализации через точный порядок и состав вызовов deepcopy.

### 3.3 Дополнительные уточнения, обязательные для спецификации

#### A. Покрытие `_validate_docx_archive` существенно недостаточно

Функция содержит 7 независимых охранных проверок:

1. размер архива больше лимита;
2. `BadZipFile`;
3. пустой архив;
4. число entries больше лимита;
5. общий uncompressed size больше лимита;
6. compression ratio больше лимита;
7. отсутствие `[Content_Types].xml`.

По факту тестами покрыта только одна ветка — oversized uncompressed archive. Остальные 6 веток должны быть закрыты отдельно и явно.

#### B. Финальная сборка в pipeline недопокрыта

Цепочка финализации результата включает:

1. `convert_markdown_to_docx_bytes`;
2. `preserve_source_paragraph_properties`;
3. `normalize_semantic_output_docx`;
4. `reinsert_inline_images`.

Нужны отдельные pipeline-контекстные тесты на crash/failure для каждого шага, а также один сильный success-path, который валидирует не только state update, но и корректность итогового DOCX как артефакта.

#### C. `test_application_flow.py` не является реальной интеграцией

Хотя файл выглядит как интеграционный слой, внутри почти все тяжёлые зависимости подменены моками. Поэтому соединение:

`prepare_run_context` → `extract_document_content_from_docx` → `build_editing_jobs`

не валидируется на реальных данных и остаётся тестовым пробелом.

#### D. Покрытие prompt registry недостаточно

Критический конфигурационный файл image prompt registry покрыт слишком бедно. Нужны негативные тесты на:

- невалидный TOML;
- отсутствие обязательных ключей;
- некорректный профиль;
- сохранность safety-constraints.

#### E. `conftest.py` недоиспользован

Фейки и SessionState-заменители дублируются в нескольких тестовых файлах. Это не баг, но источник будущего drift при изменении контракта состояния.

#### F. Live API тесты не являются реальным регрессионным барьером

Тесты в `test_real_image_pipeline.py`, завязанные на `LIVE_API_ENABLED`, вероятно почти никогда не выполняются в CI и не должны рассматриваться как часть основного защитного слоя.

## 4. Целевая модель тестового набора

После рефакторинга тестовый набор должен быть разделён концептуально на следующие слои:

### 4.1 Unit / contract

Быстрые тесты изолированных модулей, которые проверяют:

- входные и выходные контракты;
- критические guard-ветки;
- нормализацию данных;
- устойчивость к ошибкам на уровне функций.

### 4.2 Integration

Тесты, которые соединяют несколько production-модулей без избыточного мокинга и страхуют реальные бизнес-цепочки:

- DOCX extraction → segmentation → plan building;
- processing worker → event emission → runtime state update;
- document pipeline success/failure на финальной сборке;
- app-level сценарии, близкие к реальному пользовательскому пути.

### 4.3 Smoke / optional acceptance

Небольшой слой тестов для внешних скриптов, контрактов окружения и, при необходимости, live API. Эти тесты не должны подменять основной regression-suite.

### 4.4 Archive / docs guards

Spec-like проверки документации и архивных markdown-артефактов должны быть явно отделены от продуктовых регрессий и, при необходимости, выполняться отдельной джобой.

## 5. Обязательные изменения по приоритетам

## P0 — обязательный этап

### P0.1 Добавить один полноценный сквозной success-path test

Нужно добавить тест, который проходит цепочку:

1. загрузка небольшого DOCX;
2. подготовка контекста;
3. обработка через mocked AI / deterministic stub;
4. финальная сборка output DOCX;
5. валидация, что итоговый DOCX не пустой, открывается, содержит ожидаемые блоки и не теряет обязательную структуру.

Минимальное требование: тест должен проверять не только state changes, но и фактический итоговый бинарный результат.

### P0.2 Закрыть 6 непокрытых веток `_validate_docx_archive`

Добавить отдельные тесты на:

- oversized compressed archive;
- `BadZipFile`;
- empty archive;
- too many entries;
- compression ratio anomaly;
- missing `[Content_Types].xml`.

Требование: каждая ветка должна иметь собственный осмысленный тест с проверкой конкретного failure-contract, а не общий параметризованный smoke без различения причин.

### P0.3 Закрыть final-assembly crash paths в `document_pipeline`

Добавить тесты, в которых по отдельности падают:

- `convert_markdown_to_docx_bytes`;
- `preserve_source_paragraph_properties`;
- `normalize_semantic_output_docx`;
- `reinsert_inline_images`.

Для каждой ветки должны быть проверены:

- статус результата;
- error surface / failure reason;
- отсутствие ложного success;
- корректное обновление runtime state, если оно предусмотрено контрактом.

### P0.4 Привести compare-panel тесты в соответствие коду

Текущие тесты с семантикой `applies_selected_variants` и аналогичной должны быть либо:

- удалены;
- либо переименованы и переписаны под реальный контракт render-only поведения.

Недопустимо оставлять названия, обещающие apply-логику, которой больше нет в production-коде.

## P1 — важный этап после P0

### P1.1 Удалить dead code `render_image_compare_selector` и связанный тест

Если не найден актуальный production-use-case, функция должна быть удалена вместе с тестом. Если по организационным причинам удаление откладывается, необходимо:

- явно пометить её deprecated;
- исключить из дальнейшего расширения покрытия;
- создать отдельную задачу на удаление.

Предпочтительный вариант в этой спецификации: удалить dead code и его тест.

### P1.2 Усилить `test_image_prompts.py`

Добавить контрактные тесты для image prompt registry на:

- parse failure;
- missing required fields;
- invalid profile structure;
- сохранение safety / fallback требований.

### P1.3 Добавить success-path для `run_processing_worker`

Нужен тест, который подтверждает:

- успешное завершение worker;
- корректный `outcome`;
- emission ожидаемых событий;
- совместимость с последующим применением событий на runtime-слое.

### P1.4 Ослабить implementation-detail тесты в `test_preparation.py`

Тесты не должны фиксировать точное число или порядок внутренних вызовов вроде `deepcopy`, если это не является частью внешнего контракта. Их нужно переписать так, чтобы проверялось:

- отсутствие shared mutable state;
- независимость возвращаемых структур;
- корректность cache semantics.

## P2 — структурная чистка набора

### P2.1 Вынести spec-like guard тесты из основного regression-suite

Тесты, проверяющие архивные markdown/spec артефакты, должны быть:

- перемещены в отдельную категорию;
- либо запускаться отдельной CI job;
- либо переименованы так, чтобы не маскироваться под продуктовые регрессии.

### P2.2 Консолидировать SessionState и shared test fixtures

Нужно вынести в `conftest.py` или соседний test helper:

- единый fake SessionState;
- общие фабрики runtime state;
- типовые DOCX/image фикстуры, если они повторяются.

Цель: сократить дублирование и снизить цену изменения state-контракта.

### P2.3 Пересмотреть ценность `test_app_runtime.py`

Thin-wiring тесты должны быть либо сокращены до минимально необходимого набора, либо заменены более полезным integration-тестом верхнего уровня.

### P2.4 Пересмотреть роль `test_real_image_pipeline.py`

Live API тесты не должны учитываться как часть обязательного регрессионного барьера. Их следует:

- явно пометить как optional acceptance;
- исключить из критериев достаточности основного покрытия;
- запускать только в специальном окружении.

## 7. Критерии приёмки рефакторинга

Рефакторинг тестов считается успешным, если одновременно выполнены следующие условия:

1. В наборе появился хотя бы один настоящий сквозной DOCX success-path test.
2. Все 7 guard-веток `_validate_docx_archive` имеют явное тестовое покрытие.
3. Все 4 шага финальной сборки в `document_pipeline` имеют отдельные crash-path тесты.
4. В основном regression-suite отсутствуют stale compare-panel тесты старого контракта.
5. Dead code `render_image_compare_selector` и его тест удалены либо формально выведены из употребления с отдельной задачей на удаление.
6. `test_processing_service.py` содержит хотя бы один сильный success-path.
7. `test_image_prompts.py` покрывает негативные конфигурационные сценарии.
8. spec-like guard тесты больше не смешиваются с продуктовыми регрессиями.
9. Количество дублирующихся SessionState-подделок уменьшено за счёт консолидации shared fixtures.

## 8. Что не входит в эту спецификацию

Вне рамок текущего рефакторинга остаются:

- изменение production-логики, если оно не нужно для тестируемости;
- переработка image algorithmics;
- оптимизация скорости исполнения каждого теста как самостоятельная цель;
- смена общей архитектуры приложения.

## 9. Рекомендуемый порядок исполнения

1. Починить test truthfulness: compare-panel, dead code, misleading names.
2. Закрыть P0 пробелы в archive guards и final assembly.
3. Добавить сквозной DOCX happy-path.
4. Усилить success-path у processing-service и prompt registry.
5. Выполнить структурную чистку suite: spec guards, SessionState fixtures, thin-wiring tests.

## 10. Ожидаемый итог

После выполнения этой спецификации тестовый набор должен сместиться:

- от проверки внутренних деталей реализации;
- к защите реальных контрактов, пользовательских сценариев и критических failure/success точек.

Главный целевой результат: тесты должны достоверно отвечать на вопрос, может ли пользователь безопасно загрузить DOCX, пройти обработку и получить корректный итоговый документ без потери критической структуры и без скрытых регрессий в финальной сборке.
