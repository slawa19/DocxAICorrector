# Спецификация v1: сохранение и улучшение изображений в DOCX

## 1. Краткое описание

Нужно расширить текущий pipeline литературного редактирования `.docx`, чтобы приложение:

1. извлекало из исходного документа встроенные изображения;
2. сохраняло их позицию в текстовом потоке;
3. при необходимости улучшало их качество с помощью ИИ;
4. вставляло изображения обратно в итоговый `.docx` после завершения текстовой обработки.

Цель v1 — дать **простой и надежный** способ не терять картинки при текущей схеме:

`DOCX -> извлечение текста -> AI-редактирование -> Markdown -> Pandoc -> DOCX`

Приоритет v1 — **сохранность смысла и устойчивость результата**, а не максимальная полнота поддержки всех видов Word-объектов.

---

## 2. Цели v1

### Что должно уметь решение

1. Извлекать из `.docx` **inline-изображения**, встречающиеся в основном потоке документа.
2. Сохранять порядок следования изображений относительно текста.
3. Подставлять в текст служебные маркеры-плейсхолдеры, которые проходят через существующий AI-pipeline.
4. Улучшать изображение через отдельный шаг обработки, не смешивая его с текстовой редактурой.
5. После сборки итогового `.docx` находить плейсхолдеры и заменять их обратно на изображения.
6. Если улучшение изображения не удалось, вставлять **оригинальное изображение**, а не прерывать весь процесс.

### Что не входит в v1

- Полная поддержка floating shapes, SmartArt, OLE-объектов и произвольной сложной Word-верстки.
- Гарантированное сохранение исходных anchors, wrapping-настроек и всех layout-свойств Word.
- "Творческая" перерисовка схем и графиков с изменением композиции.
- Распознавание и исправление текста внутри картинок как отдельная OCR-задача.

---

## 3. Основная идея решения

Для минимального вмешательства в существующую архитектуру используется подход:

### `placeholder-based media pipeline`

1. При чтении `.docx` система извлекает не только абзацы, но и изображения.
2. На месте каждого изображения в текстовый поток вставляется стабильный плейсхолдер вида:

```text
[[DOCX_IMAGE_img_001]]
```

3. Текст с плейсхолдерами проходит через существующий pipeline редактирования.
4. После сборки итогового `.docx` через Pandoc документ открывается повторно.
5. Плейсхолдеры находятся в тексте и заменяются на соответствующие изображения.

Такой подход позволяет:

- не переписывать текущую логику обработки текста;
- не отказываться от Markdown и Pandoc;
- сохранить изображения в итоговом документе;
- внедрить улучшение изображений как отдельный, изолированный этап.

---

## 4. Пользовательский сценарий v1

1. Пользователь загружает `.docx`, где есть текст и изображения.
2. Система извлекает текст и изображения, сохраняя их порядок.
3. Текстовые блоки отправляются в ИИ на литературное редактирование как и сейчас.
4. Изображения отдельно проходят безопасное улучшение качества.
5. Из отредактированного Markdown собирается новый `.docx`.
6. В результирующем `.docx` плейсхолдеры заменяются на улучшенные изображения.
7. Если для конкретного изображения улучшение не удалось, вставляется исходная версия.
8. Пользователь скачивает итоговый `.docx`, где текст отредактирован, а изображения сохранены.

---

## 5. Pipeline v1

```text
Исходный DOCX
    -> Извлечение абзацев и inline-изображений
    -> Замена изображений на текстовые плейсхолдеры
    -> Сборка смысловых блоков
    -> AI-редактирование текста
    -> Склейка Markdown
    -> Pandoc -> DOCX
    -> Поиск плейсхолдеров в итоговом DOCX
    -> Вставка улучшенных или оригинальных изображений
    -> Готовый DOCX
```

---

## 6. Требования к плейсхолдерам

### Формат

Плейсхолдер должен быть:

- ASCII-совместимым;
- устойчивым к markdown-конвертации;
- легко находимым в итоговом тексте;
- маловероятным для случайного появления в пользовательском документе.

Рекомендуемый формат:

```text
[[DOCX_IMAGE_img_001]]
```

### Правила для модели

Системный prompt и/или user prompt должны явно запрещать модели:

- удалять плейсхолдеры;
- изменять их текст;
- перемещать их без необходимости;
- превращать их в обычный пользовательский текст.

Модель должна рассматривать такие маркеры как **служебные токены**, которые нужно вернуть без изменений.

---

## 7. Обработка изображений

### 7.1. Какие изображения поддерживаются в v1

Поддерживаются в первую очередь:

- изображения, вставленные как обычные inline pictures в основном теле документа;
- форматы, которые можно извлечь из DOCX media part и затем снова вставить в Word-документ.

### 7.2. Что считается "улучшением" в v1

В v1 используется **safe enhancement**, то есть улучшение качества **без изменения смысла**:

- повышение резкости;
- удаление шумов и артефактов сжатия;
- мягкое улучшение контраста;
- повышение читаемости деталей;
- при необходимости апскейл без изменения композиции.

### 7.3. Что запрещено в v1

Для графиков, диаграмм, схем, скриншотов и изображений с текстом запрещено:

- менять подписи;
- менять цифры, оси, легенды;
- удалять элементы;
- дорисовывать новые смысловые объекты;
- перестраивать композицию "как красивее".

То есть для таких изображений допускается только мягкое улучшение качества, а не генеративная перерисовка.

---

## 8. Правила безопасности и сохранения смысла

1. Если тип изображения не удается надежно классифицировать, применяется только безопасное улучшение или используется оригинал.
2. Если результат AI-обработки невалиден, пустой или визуально подозрительный по техническим признакам, используется оригинал.
3. Ошибка на одном изображении не должна ломать обработку всего документа.
4. Итоговый документ должен собираться даже при полном отключении image-enhancement шага.

Главное правило v1:

> Лучше вернуть исходное изображение, чем вставить "красивое", но искажающее смысл.

---

## 9. Минимальная модель данных

Для каждого изображения в pipeline достаточно хранить структуру со следующими полями:

- `image_id` — стабильный идентификатор (`img_001`, `img_002`, ...);
- `placeholder` — текст плейсхолдера;
- `original_bytes` — исходные байты изображения;
- `enhanced_bytes` — улучшенная версия, если обработка удалась;
- `filename` или `extension`;
- `position_index` — порядковая позиция в потоке документа;
- `width` / `height` — если доступны и нужны для обратной вставки;
- `status` — `original`, `enhanced`, `failed`, `unsupported`.

Этого достаточно для v1 без введения сложной иерархии классов.

---

## 10. Точки интеграции в текущую архитектуру

### `document.py`

Добавляется логика:

- извлечения inline-изображений из DOCX;
- генерации плейсхолдеров;
- включения плейсхолдеров в поток текста рядом с соответствующей позицией.

### `generation.py`

Добавляется логика:

- безопасного улучшения изображения через отдельный вызов;
- post-processing итогового `.docx` после Pandoc;
- поиска плейсхолдеров и замены их на изображения.

### `app.py`

Добавляется оркестрация шагов:

1. extract text + images;
2. process text;
3. build output docx;
4. reinsert images.

Текущая логика редактирования текстовых блоков должна остаться максимально неизменной.

---

## 11. Ограничения v1

1. Поддержка ориентирована на **основной поток документа**, а не на все возможные контейнеры Word.
2. Изображение может быть вставлено обратно с близким, но не обязательно идентичным layout-поведением.
3. Сложные Office chart objects могут потребовать отдельной обработки в v2.
4. Таблицы, сноски, колонтитулы и иные сложные элементы в рамках этой спецификации не расширяются автоматически.

---

## 12. Критерии приемки v1

Решение считается реализованным, если:

1. документ с inline-изображениями после обработки не теряет эти изображения полностью;
2. изображения возвращаются в итоговый `.docx` в корректном порядке;
3. текстовый pipeline продолжает работать как раньше;
4. плейсхолдеры не попадают в финальный документ как пользовательский текст;
5. при сбое улучшения для конкретной картинки используется оригинал;
6. графики и схемы не подвергаются опасной семантической перерисовке по умолчанию.

---

## 13. Порядок внедрения

### Этап 1

Сначала реализовать:

- извлечение изображений;
- генерацию плейсхолдеров;
- обратную вставку **без AI-улучшения**.

Это дает базовую гарантию сохранения изображений.

### Этап 2

После этого добавить:

- отдельный шаг safe enhancement;
- fallback на оригинал;
- минимальное логирование статуса по изображениям.

### Этап 3

Только после стабилизации v1 обсуждать v2:

- расширение поддержки сложных Word-объектов;
- более точное сохранение layout;
- дополнительные режимы обработки изображений.

---

## 14. Итоговое решение для v1

Для текущего проекта оптимальным считается следующий вариант:

- **не менять** базовый текстовый AI-pipeline;
- **добавить** placeholder-based обработку изображений;
- **улучшать** изображения только в безопасном режиме;
- **всегда иметь fallback** на оригинал;
- **не обещать** полную поддержку всей графики Word в первой версии.

Это самое простое решение, которое можно реализовать с минимальным риском для уже работающей архитектуры.

---

## 15. Детализация доработки по текущим модулям

Ниже описана **не новая архитектура**, а точечное расширение уже существующих модулей:

- `document.py` — читает `.docx` и собирает `ParagraphUnit`;
- `models.py` — хранит простые dataclass-модели;
- `generation.py` — вызывает модель и собирает итоговый `.docx` через Pandoc;
- `app.py` — оркестрирует pipeline.

Цель v1 — встроить поддержку изображений так, чтобы:

1. не ломать существующий контракт текстовой обработки;
2. не переписывать логику блочной редактуры;
3. свести изменения к добавлению новых функций и минимальному расширению существующих.

### 15.1. `document.py`

В текущем состоянии модуль работает на уровне `document.paragraphs` и читает только `paragraph.text`.
Этого недостаточно для изображений, потому что inline picture живет внутри `paragraph.runs`
и не попадает в `.text`.

В v1 предлагается следующий контракт:

- существующую функцию `extract_paragraph_units_from_docx(uploaded_file)` сохранить;
- добавить рядом новую функцию верхнего уровня, например:

```python
def extract_document_content_from_docx(uploaded_file) -> tuple[list[ParagraphUnit], list["ImageAsset"]]:
    ...
```

Она должна:

1. открыть документ через `Document(uploaded_file)`;
2. пройти по абзацам в исходном порядке;
3. внутри каждого абзаца пройти по `runs` в исходном порядке;
4. собрать текст и встроенные изображения в один линейный поток;
5. вернуть:
   - итоговые `ParagraphUnit`, где на месте изображений стоят плейсхолдеры;
   - список `ImageAsset` с метаданными для обратной вставки.

### 15.2. Правило сериализации абзаца

Для каждого абзаца действует следующее правило:

- если абзац содержит только текст — поведение остается текущим;
- если абзац содержит только изображение — создается `ParagraphUnit` с текстом-плейсхолдером;
- если абзац содержит и текст, и изображения — текст и плейсхолдеры собираются в одну строку
  в порядке исходных `runs`.

Пример:

```text
До картинки [img] после картинки
```

должен сериализоваться как:

```text
До картинки [[DOCX_IMAGE_img_001]] после картинки
```

Это критично для сохранения позиции inline-изображения относительно текста.

### 15.3. `models.py`

Текущие модели:

- `ParagraphUnit(text, role)`
- `DocumentBlock(paragraphs)`

Для v1 достаточно добавить еще одну простую dataclass-модель:

```python
@dataclass
class ImageAsset:
    image_id: str
    placeholder: str
    original_bytes: bytes
    enhanced_bytes: bytes | None
    extension: str
    position_index: int
    width_emu: int | None = None
    height_emu: int | None = None
    status: str = "original"
```

Дополнительно можно ввести компактную строковую конвенцию статусов:

- `original`
- `enhanced`
- `failed`
- `unsupported`

Новая модель должна использоваться только как transport-объект между извлечением,
улучшением и reinsertion, без лишней бизнес-логики.

### 15.4. `generation.py`

Текущий модуль уже умеет:

- нормализовать ответ модели;
- повторять retryable API-запросы;
- конвертировать Markdown в `.docx`.

В v1 в него предлагается добавить отдельные функции:

```python
def enhance_image_asset(image: ImageAsset) -> ImageAsset:
    ...

def reinsert_images_into_docx(docx_bytes: bytes, image_assets: list[ImageAsset]) -> bytes:
    ...
```

Их обязанности:

- `enhance_image_asset`:
  - принимает один `ImageAsset`;
  - либо возвращает его же без изменений;
  - либо записывает улучшенную версию в `enhanced_bytes` и меняет `status`;
  - при ошибке не выбрасывает фатальную ошибку наружу, а переводит объект в fallback-состояние.

- `reinsert_images_into_docx`:
  - открывает результат Pandoc через `python-docx`;
  - находит текстовые плейсхолдеры в абзацах;
  - заменяет плейсхолдеры на изображения;
  - возвращает итоговые байты `.docx`.

### 15.5. `app.py`

Оркестрация в `app.py` должна быть расширена минимально:

1. чтение документа меняется с `extract_paragraph_units_from_docx(...)`
   на новую функцию, возвращающую `paragraphs` и `image_assets`;
2. дальше текстовый pipeline (`build_semantic_blocks`, `build_editing_jobs`,
   `generate_markdown_block`) работает как раньше;
3. после `convert_markdown_to_docx_bytes(...)` вызывается reinsertion-шаг;
4. если image-шаг не удался частично, итоговый документ все равно отдается пользователю.

## 16. Низкоуровневый алгоритм извлечения изображений

### 16.1. Что считаем поддерживаемым в v1

Поддерживаемыми считаются изображения:

- находящиеся в основном потоке документа;
- представленные как inline drawing внутри `run`;
- имеющие доступный binary-part внутри DOCX-пакета.

### 16.2. Алгоритм прохода

Алгоритм извлечения должен быть таким:

1. пройти по каждому `paragraph` документа по порядку;
2. внутри абзаца пройти по `runs` по порядку;
3. для каждого `run`:
   - добавить его текстовую часть, если она есть;
   - проверить наличие drawing/picture-узла в XML;
   - если в `run` найдено изображение, создать `ImageAsset`;
   - в поток текста вставить соответствующий `placeholder`.

### 16.3. Требование к порядку

`position_index` должен отражать **не просто номер картинки в документе**,
а ее положение в линейном потоке контента. Это нужно, чтобы:

- корректно логировать ход обработки;
- отлаживать сбои reinsertion;
- в будущем поддержать более точную повторную вставку.

### 16.4. Формат идентификатора

Идентификатор генерируется строго последовательно:

```text
img_001
img_002
img_003
```

Соответствующий плейсхолдер:

```text
[[DOCX_IMAGE_img_001]]
```

Формат должен быть стабильным во всех модулях.

## 17. Детализация reinsertion после Pandoc

В v1 повторная вставка выполняется **после** `convert_markdown_to_docx_bytes(...)`,
а не во время генерации Markdown.

### 17.1. Почему именно post-processing

Это решение выбрано потому, что:

- текущий pipeline уже стабильно собирает `.docx` через Pandoc;
- не нужно переводить изображения в markdown-ссылки и управлять временными файлами;
- reinsertion можно изолировать в одном месте;
- fallback на оригинал проще реализовать на уровне `python-docx`.

### 17.2. Контракт reinsertion-функции

Функция `reinsert_images_into_docx(...)` должна:

1. принять `docx_bytes` после Pandoc;
2. открыть документ через `Document(...)`;
3. найти все плейсхолдеры `[[DOCX_IMAGE_*]]` в основном потоке;
4. для каждого найденного маркера:
   - выбрать `enhanced_bytes`, если они есть и валидны;
   - иначе выбрать `original_bytes`;
5. вставить изображение максимально близко к позиции плейсхолдера;
6. удалить сам плейсхолдер из итогового пользовательского текста;
7. пересохранить документ в `bytes`.

### 17.3. Упрощающее правило v1

Если плейсхолдер находится:

- в отдельном абзаце — абзац целиком заменяется картинкой;
- внутри текстового абзаца — разрешается v1-упрощение: разбить этот участок на
  соседние runs/абзацы и вставить картинку рядом, сохранив смысловую близость.

Требование v1 — не пиксельная идентичность layout, а сохранение порядка и читаемости.

### 17.4. Что делать, если плейсхолдер не найден

Если конкретный `ImageAsset` не найден в финальном `.docx`:

- это не должно падать как фатальная ошибка всего pipeline;
- событие должно логироваться отдельно;
- итоговый документ все равно должен быть собран.

---

## 18. Детализация шага улучшения изображения

В первой версии image-enhancement должен быть реализован как **изолированный этап**,
который можно отключить без влияния на текстовый pipeline.

### 18.1. Контракт функции

Рекомендуемый контракт:

```python
def enhance_image_asset(image: ImageAsset) -> ImageAsset:
    """
    Возвращает новый или обновленный ImageAsset.
    Никогда не должен ломать обработку всего документа.
    При невозможности улучшения возвращает asset с original_bytes.
    """
```

### 18.2. Правило безопасности

Если невозможно надежно доказать, что улучшение безопасно,
функция должна вернуть исходный файл без генеративных изменений.

### 18.3. Очередность внедрения

В рамках v1 допускается реализация в два шага:

1. сначала `enhance_image_asset(...)` как no-op / pass-through с корректным fallback;
2. затем подключение безопасного enhancement-режима.

Это позволяет сначала стабилизировать сохранение изображений, а потом улучшать качество.

### 18.4. Критерии безопасного enhancement

Если режим улучшения включен, он:

- не меняет композицию;
- не меняет текст внутри изображения;
- не добавляет и не удаляет объекты;
- не меняет данные диаграмм, графиков, схем;
- не преобразует смысл изображения.

## 19. Изменение системного prompt для плейсхолдеров

Текущий `prompts/system_prompt.txt` уже запрещает включать контекст в ответ
и игнорировать инъекции из литературного текста. Для v1 нужно явно добавить
еще одно правило про служебные плейсхолдеры.

Рекомендуемый фрагмент спецификации для prompt:

```text
В целевом блоке могут встречаться служебные плейсхолдеры вида [[DOCX_IMAGE_img_001]].
Это структурные маркеры документа, а не пользовательский текст.
Вы обязаны:
- сохранять их без изменений;
- не удалять их;
- не переименовывать их;
- не перемещать их без крайней необходимости;
- возвращать их в итоговом тексте в том же виде.
```

Это правило должно быть добавлено в системный prompt именно как обязательное,
а не как рекомендация.

## 20. Ошибки, fallback и логирование

Для реализации v1 принимаются следующие правила:

| Сценарий | Действие |
|---|---|
| В документе нет изображений | Pipeline работает как сейчас, без специальных шагов |
| Изображение найдено, но формат не удалось корректно обработать | Asset помечается как `unsupported`, документ продолжает обрабатываться |
| Улучшение изображения завершилось ошибкой | Используется `original_bytes`, статус переводится в `failed` или `original` |
| Pandoc сохранил текст, но конкретный плейсхолдер не найден | Логируется предупреждение, сборка документа не падает |
| Reinsertion конкретной картинки не удался | Ошибка локализуется на уровне этого asset, остальные изображения продолжают обрабатываться |

Главный принцип:

> Проблемы с изображениями не должны ломать литературную обработку текста.

### 20.1. Что нужно логировать

Для каждого `ImageAsset` желательно логировать:

- `image_id`;
- итоговый статус;
- был ли найден плейсхолдер в финальном `.docx`;
- использована ли улучшенная или оригинальная версия;
- краткую причину fallback при ошибке.

## 21. Тестовые сценарии для реализации

Текущая спецификация описывает доработку; при реализации должны быть добавлены
точечные тесты именно на новые контракты.

### 21.1. Unit-тесты

1. Извлечение DOCX без картинок:
   - возвращаются обычные `ParagraphUnit`;
   - список `ImageAsset` пуст.

2. Извлечение DOCX с одной картинкой между абзацами:
   - создается один `ImageAsset`;
   - в тексте появляется правильный плейсхолдер.

3. Извлечение DOCX с текстом и картинкой в одном абзаце:
   - порядок текста и плейсхолдера сохраняется.

4. Fallback при ошибке enhancement:
   - функция не выбрасывает фатальную ошибку наружу;
   - используется оригинальная версия.

5. Reinsertion удаляет плейсхолдер из финального текста:
   - в итоговом `.docx` не остается `[[DOCX_IMAGE_*]]`.

### 21.2. Интеграционные тесты

1. DOCX с 2–3 изображениями проходит полный pipeline:
   - текст обрабатывается;
   - итоговый `.docx` собирается;
   - изображения присутствуют в корректном порядке.

2. DOCX, где одно изображение "ломается" на enhancement-шаге:
   - итоговый документ все равно создается;
   - проблемная картинка берется в оригинале.

### 21.3. Специальная проверка на устойчивость плейсхолдера

Отдельно должна быть проверена гипотеза, что строка:

```text
[[DOCX_IMAGE_img_001]]
```

проходит через:

- текстовую редактуру моделью;
- сборку Markdown;
- конвертацию через Pandoc;

без разрушения формата, мешающего последующему поиску.

Если окажется, что Pandoc или модель системно искажает такой формат,
в реализации v1 должен быть выбран другой устойчивый маркер,
но с сохранением всей описанной архитектуры.

## 22. Итоговая реализационная декомпозиция

Для фактической разработки порядок работ должен быть таким:

1. добавить `ImageAsset` в `models.py`;
2. добавить извлечение текста + изображений в `document.py`;
3. убедиться, что placeholder-формат устойчив на тестовом round-trip;
4. добавить reinsertion в `generation.py`;
5. подключить orchestration в `app.py`;
6. только после этого включать enhancement-шаг;
7. затем добавить targeted tests на fallback и корректный порядок изображений.

Такой порядок минимизирует риск: сначала сохраняем картинки как есть,
потом аккуратно улучшаем их качество.
