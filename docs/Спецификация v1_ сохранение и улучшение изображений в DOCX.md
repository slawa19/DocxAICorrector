# Спецификация v1-v2: image safe-mode и image semantic-redraw-mode для изображений в DOCX

## 0. Переименование режимов и предварительное предложение по UI/архитектуре

### 0.1. Новые названия режимов

В этой редакции документа фиксируются следующие названия:

- текущая спецификация v1 называется **image safe-mode**;
- новая расширенная спецификация v2 называется **image semantic-redraw-mode**.

Под **image safe-mode** понимается режим, в котором изображение:

- сохраняется в документе;
- может быть мягко улучшено;
- не должно подвергаться генеративной семантической перерисовке.

Под **image semantic-redraw-mode** понимается режим, в котором система:

- анализирует изображение;
- пытается понять его семантическую структуру;
- при подходящем типе изображения перестраивает его в более чистом, читаемом и современном виде;
- при этом обязана сохранять исходный смысл, структуру и логические связи.

### 0.2. Предложение по UI

На основе текущего интерфейса Streamlit оптимальным считается минимальное расширение sidebar.

Предлагается добавить новый пользовательский блок настроек:

```text
Режим обработки изображений
(
  • image safe-mode
  • image semantic-redraw-mode
)
```

Рекомендуемое поведение UI:

1. По умолчанию выбран `image safe-mode`.
2. При выборе `image semantic-redraw-mode` интерфейс показывает предупреждение:
   - режим может менять визуальное оформление изображения сильнее, чем safe-mode;
   - система будет стараться сохранить смысл, но для некоторых типов изображений может откатиться к safe-mode.
3. В этом же блоке допустимо добавить подпись:
   - `Prompt для semantic redraw выбирается автоматически после анализа изображения`.
4. В результатах обработки и/или run-log желательно показывать по каждому изображению:
   - определенный тип изображения;
   - выбранный prompt-профиль;
   - выбранную стратегию (`safe`, `semantic_redraw_direct`, `semantic_redraw_structured`);
   - итоговый статус (`enhanced`, `redrawn`, `fallback_original`, `fallback_safe`).

Для текущей структуры приложения логичнее всего встроить этот элемент в `ui.render_sidebar()`
рядом с выбором модели, размера блока и retry.

### 0.3. Предложение по архитектуре

Для v2 уже недостаточно одной функции "улучшить картинку", поэтому архитектуру лучше
разделить на три самостоятельные части:

1. **image extraction / reinsertion**
   - извлечение изображений и возврат их в итоговый `.docx`;
2. **image analysis**
   - определение типа изображения;
   - оценка, допустим ли semantic redraw;
   - выбор prompt-профиля и стратегии рендеринга;
3. **image generation**
   - safe enhancement;
   - semantic redraw через `gpt-image-1`;
   - structured redraw через промежуточное JSON/Mermaid/SVG-представление.

С точки зрения модулей это можно выразить так:

- `document.py` — извлечение текста, плейсхолдеров и `ImageAsset`;
- `generation.py` — текстовая генерация и общая сборка результата;
- новый `image_analysis.py` — анализ изображения и выбор стратегии;
- новый `image_prompts.py` или `config.py`-слой — реестр prompt-профилей по типам изображений;
- новый `image_generation.py` — safe enhancement, semantic redraw и structured redraw;
- `app.py` — оркестрация режима, выбранного пользователем в UI.

Это лучше, чем перегружать `generation.py` всей image-логикой v2, потому что semantic redraw
включает классификацию, выбор prompt’а, выбор модели и fallback-ветвления.

## 1. Краткое описание

Нужно расширить текущий pipeline литературного редактирования `.docx`, чтобы приложение:

1. извлекало из исходного документа встроенные изображения;
2. сохраняло их позицию в текстовом потоке;
3. при необходимости улучшало их качество с помощью ИИ;
4. вставляло изображения обратно в итоговый `.docx` после завершения текстовой обработки.

Цель v1 / image safe-mode — дать **простой и надежный** способ не терять картинки при текущей схеме:

`DOCX -> извлечение текста -> AI-редактирование -> Markdown -> Pandoc -> DOCX`

Приоритет v1 — **сохранность смысла и устойчивость результата**, а не максимальная полнота поддержки всех видов Word-объектов.

---

## 2. Цели v1

### Что должно уметь решение

1. Извлекать из `.docx` **inline-изображения**, встречающиеся в основном потоке документа.
2. Сохранять порядок следования изображений относительно текста.
3. Подставлять в текст служебные маркеры-плейсхолдеры, которые проходят через существующий AI-pipeline.
4. Улучшать изображение через отдельный шаг обработки, не смешивая его с текстовой редактурой.
5. После сборки итогового `.docx` находить плейсхолдеры и заменять их обратно на изображения.
6. Если улучшение изображения не удалось, вставлять **оригинальное изображение**, а не прерывать весь процесс.

### Что не входит в v1

- Полная поддержка floating shapes, SmartArt, OLE-объектов и произвольной сложной Word-верстки.
- Гарантированное сохранение исходных anchors, wrapping-настроек и всех layout-свойств Word.
- "Творческая" перерисовка схем и графиков с изменением композиции.
- Распознавание и исправление текста внутри картинок как отдельная OCR-задача.

---

## 3. Основная идея решения

Для минимального вмешательства в существующую архитектуру используется подход:

### `placeholder-based media pipeline`

1. При чтении `.docx` система извлекает не только абзацы, но и изображения.
2. На месте каждого изображения в текстовый поток вставляется стабильный плейсхолдер вида:

```text
[[DOCX_IMAGE_img_001]]
```

3. Текст с плейсхолдерами проходит через существующий pipeline редактирования.
4. После сборки итогового `.docx` через Pandoc документ открывается повторно.
5. Плейсхолдеры находятся в тексте и заменяются на соответствующие изображения.

Такой подход позволяет:

- не переписывать текущую логику обработки текста;
- не отказываться от Markdown и Pandoc;
- сохранить изображения в итоговом документе;
- внедрить улучшение изображений как отдельный, изолированный этап.

---

## 4. Пользовательский сценарий v1

1. Пользователь загружает `.docx`, где есть текст и изображения.
2. Система извлекает текст и изображения, сохраняя их порядок.
3. Текстовые блоки отправляются в ИИ на литературное редактирование как и сейчас.
4. Изображения отдельно проходят безопасное улучшение качества.
5. Из отредактированного Markdown собирается новый `.docx`.
6. В результирующем `.docx` плейсхолдеры заменяются на улучшенные изображения.
7. Если для конкретного изображения улучшение не удалось, вставляется исходная версия.
8. Пользователь скачивает итоговый `.docx`, где текст отредактирован, а изображения сохранены.

---

## 5. Pipeline v1

```text
Исходный DOCX
    -> Извлечение абзацев и inline-изображений
    -> Замена изображений на текстовые плейсхолдеры
    -> Сборка смысловых блоков
    -> AI-редактирование текста
    -> Склейка Markdown
    -> Pandoc -> DOCX
    -> Поиск плейсхолдеров в итоговом DOCX
    -> Вставка улучшенных или оригинальных изображений
    -> Готовый DOCX
```

---

## 6. Требования к плейсхолдерам

### Формат

Плейсхолдер должен быть:

- ASCII-совместимым;
- устойчивым к markdown-конвертации;
- легко находимым в итоговом тексте;
- маловероятным для случайного появления в пользовательском документе.

Рекомендуемый формат:

```text
[[DOCX_IMAGE_img_001]]
```

### Правила для модели

Системный prompt и/или user prompt должны явно запрещать модели:

- удалять плейсхолдеры;
- изменять их текст;
- перемещать их без необходимости;
- превращать их в обычный пользовательский текст.

Модель должна рассматривать такие маркеры как **служебные токены**, которые нужно вернуть без изменений.

---

## 7. Обработка изображений

### 7.1. Какие изображения поддерживаются в v1

Поддерживаются в первую очередь:

- изображения, вставленные как обычные inline pictures в основном теле документа;
- форматы, которые можно извлечь из DOCX media part и затем снова вставить в Word-документ.

### 7.2. Что считается "улучшением" в v1

В v1 используется **safe enhancement**, то есть улучшение качества **без изменения смысла**:

- повышение резкости;
- удаление шумов и артефактов сжатия;
- мягкое улучшение контраста;
- повышение читаемости деталей;
- при необходимости апскейл без изменения композиции.

### 7.3. Что запрещено в v1

Для графиков, диаграмм, схем, скриншотов и изображений с текстом запрещено:

- менять подписи;
- менять цифры, оси, легенды;
- удалять элементы;
- дорисовывать новые смысловые объекты;
- перестраивать композицию "как красивее".

То есть для таких изображений допускается только мягкое улучшение качества, а не генеративная перерисовка.

---

## 8. Правила безопасности и сохранения смысла

1. Если тип изображения не удается надежно классифицировать, применяется только безопасное улучшение или используется оригинал.
2. Если результат AI-обработки невалиден, пустой или визуально подозрительный по техническим признакам, используется оригинал.
3. Ошибка на одном изображении не должна ломать обработку всего документа.
4. Итоговый документ должен собираться даже при полном отключении image-enhancement шага.

Главное правило v1:

> Лучше вернуть исходное изображение, чем вставить "красивое", но искажающее смысл.

---

## 9. Минимальная модель данных

Для каждого изображения в pipeline достаточно хранить структуру со следующими полями:

- `image_id` — стабильный идентификатор (`img_001`, `img_002`, ...);
- `placeholder` — текст плейсхолдера;
- `original_bytes` — исходные байты изображения;
- `enhanced_bytes` — улучшенная версия, если обработка удалась;
- `filename` или `extension`;
- `position_index` — порядковая позиция в потоке документа;
- `width` / `height` — если доступны и нужны для обратной вставки;
- `status` — `original`, `enhanced`, `failed`, `unsupported`.

Этого достаточно для v1 без введения сложной иерархии классов.

---

## 10. Точки интеграции в текущую архитектуру

### `document.py`

Добавляется логика:

- извлечения inline-изображений из DOCX;
- генерации плейсхолдеров;
- включения плейсхолдеров в поток текста рядом с соответствующей позицией.

### `generation.py`

Добавляется логика:

- безопасного улучшения изображения через отдельный вызов;
- post-processing итогового `.docx` после Pandoc;
- поиска плейсхолдеров и замены их на изображения.

### `app.py`

Добавляется оркестрация шагов:

1. extract text + images;
2. process text;
3. build output docx;
4. reinsert images.

Текущая логика редактирования текстовых блоков должна остаться максимально неизменной.

---

## 11. Ограничения v1

1. Поддержка ориентирована на **основной поток документа**, а не на все возможные контейнеры Word.
2. Изображение может быть вставлено обратно с близким, но не обязательно идентичным layout-поведением.
3. Сложные Office chart objects могут потребовать отдельной обработки в v2.
4. Таблицы, сноски, колонтитулы и иные сложные элементы в рамках этой спецификации не расширяются автоматически.

---

## 12. Критерии приемки v1

Решение считается реализованным, если:

1. документ с inline-изображениями после обработки не теряет эти изображения полностью;
2. изображения возвращаются в итоговый `.docx` в корректном порядке;
3. текстовый pipeline продолжает работать как раньше;
4. плейсхолдеры не попадают в финальный документ как пользовательский текст;
5. при сбое улучшения для конкретной картинки используется оригинал;
6. графики и схемы не подвергаются опасной семантической перерисовке по умолчанию.

---

## 13. Порядок внедрения

### Этап 1

Сначала реализовать:

- извлечение изображений;
- генерацию плейсхолдеров;
- обратную вставку **без AI-улучшения**.

Это дает базовую гарантию сохранения изображений.

### Этап 2

После этого добавить:

- отдельный шаг safe enhancement;
- fallback на оригинал;
- минимальное логирование статуса по изображениям.

### Этап 3

Только после стабилизации v1 обсуждать v2:

- расширение поддержки сложных Word-объектов;
- более точное сохранение layout;
- дополнительные режимы обработки изображений.

---

## 14. Итоговое решение для v1

Для текущего проекта оптимальным считается следующий вариант:

- **не менять** базовый текстовый AI-pipeline;
- **добавить** placeholder-based обработку изображений;
- **улучшать** изображения только в безопасном режиме;
- **всегда иметь fallback** на оригинал;
- **не обещать** полную поддержку всей графики Word в первой версии.

Это самое простое решение, которое можно реализовать с минимальным риском для уже работающей архитектуры.

## 15. Детализация доработки по текущим модулям

Ниже описана **не новая архитектура**, а точечное расширение уже существующих модулей:

- `document.py` — читает `.docx` и собирает `ParagraphUnit`;
- `models.py` — хранит простые dataclass-модели;
- `generation.py` — вызывает модель и собирает итоговый `.docx` через Pandoc;
- `app.py` — оркестрирует pipeline.

Цель v1 — встроить поддержку изображений так, чтобы:

1. не ломать существующий контракт текстовой обработки;
2. не переписывать логику блочной редактуры;
3. свести изменения к добавлению новых функций и минимальному расширению существующих.

### 15.1. `document.py`

В текущем состоянии модуль работает на уровне `document.paragraphs` и читает только `paragraph.text`.
Этого недостаточно для изображений, потому что inline picture живет внутри `paragraph.runs`
и не попадает в `.text`.

В v1 предлагается следующий контракт:

- существующую функцию `extract_paragraph_units_from_docx(uploaded_file)` сохранить;
- добавить рядом новую функцию верхнего уровня, например:

```python
def extract_document_content_from_docx(uploaded_file) -> tuple[list[ParagraphUnit], list["ImageAsset"]]:
    ...
```

Она должна:

1. открыть документ через `Document(uploaded_file)`;
2. пройти по абзацам в исходном порядке;
3. внутри каждого абзаца пройти по `runs` в исходном порядке;
4. собрать текст и встроенные изображения в один линейный поток;
5. вернуть:
   - итоговые `ParagraphUnit`, где на месте изображений стоят плейсхолдеры;
   - список `ImageAsset` с метаданными для обратной вставки.

### 15.2. Правило сериализации абзаца

Для каждого абзаца действует следующее правило:

- если абзац содержит только текст — поведение остается текущим;
- если абзац содержит только изображение — создается `ParagraphUnit` с текстом-плейсхолдером;
- если абзац содержит и текст, и изображения — текст и плейсхолдеры собираются в одну строку
  в порядке исходных `runs`.

Пример:

```text
До картинки [img] после картинки
```

должен сериализоваться как:

```text
До картинки [[DOCX_IMAGE_img_001]] после картинки
```

Это критично для сохранения позиции inline-изображения относительно текста.

### 15.3. `models.py`

Текущие модели:

- `ParagraphUnit(text, role)`
- `DocumentBlock(paragraphs)`

Для v1 достаточно добавить еще одну простую dataclass-модель:

```python
@dataclass
class ImageAsset:
    image_id: str
    placeholder: str
    original_bytes: bytes
    enhanced_bytes: bytes | None
    extension: str
    position_index: int
    width_emu: int | None = None
    height_emu: int | None = None
    status: str = "original"
```

Дополнительно можно ввести компактную строковую конвенцию статусов:

- `original`
- `enhanced`
- `failed`
- `unsupported`

Новая модель должна использоваться только как transport-объект между извлечением,
улучшением и reinsertion, без лишней бизнес-логики.

### 15.4. `generation.py`

Текущий модуль уже умеет:

- нормализовать ответ модели;
- повторять retryable API-запросы;
- конвертировать Markdown в `.docx`.

В v1 в него предлагается добавить отдельные функции:

```python
def enhance_image_asset(image: ImageAsset) -> ImageAsset:
    ...

def reinsert_images_into_docx(docx_bytes: bytes, image_assets: list[ImageAsset]) -> bytes:
    ...
```

Их обязанности:

- `enhance_image_asset`:
  - принимает один `ImageAsset`;
  - либо возвращает его же без изменений;
  - либо записывает улучшенную версию в `enhanced_bytes` и меняет `status`;
  - при ошибке не выбрасывает фатальную ошибку наружу, а переводит объект в fallback-состояние.

- `reinsert_images_into_docx`:
  - открывает результат Pandoc через `python-docx`;
  - находит текстовые плейсхолдеры в абзацах;
  - заменяет плейсхолдеры на изображения;
  - возвращает итоговые байты `.docx`.

### 15.5. `app.py`

Оркестрация в `app.py` должна быть расширена минимально:

1. чтение документа меняется с `extract_paragraph_units_from_docx(...)`
   на новую функцию, возвращающую `paragraphs` и `image_assets`;
2. дальше текстовый pipeline (`build_semantic_blocks`, `build_editing_jobs`,
   `generate_markdown_block`) работает как раньше;
3. после `convert_markdown_to_docx_bytes(...)` вызывается reinsertion-шаг;
4. если image-шаг не удался частично, итоговый документ все равно отдается пользователю.

## 16. Низкоуровневый алгоритм извлечения изображений

### 16.1. Что считаем поддерживаемым в v1

Поддерживаемыми считаются изображения:

- находящиеся в основном потоке документа;
- представленные как inline drawing внутри `run`;
- имеющие доступный binary-part внутри DOCX-пакета.

### 16.2. Алгоритм прохода

Алгоритм извлечения должен быть таким:

1. пройти по каждому `paragraph` документа по порядку;
2. внутри абзаца пройти по `runs` по порядку;
3. для каждого `run`:
   - добавить его текстовую часть, если она есть;
   - проверить наличие drawing/picture-узла в XML;
   - если в `run` найдено изображение, создать `ImageAsset`;
   - в поток текста вставить соответствующий `placeholder`.

### 16.3. Требование к порядку

`position_index` должен отражать **не просто номер картинки в документе**,
а ее положение в линейном потоке контента. Это нужно, чтобы:

- корректно логировать ход обработки;
- отлаживать сбои reinsertion;
- в будущем поддержать более точную повторную вставку.

### 16.4. Формат идентификатора

Идентификатор генерируется строго последовательно:

```text
img_001
img_002
img_003
```

Соответствующий плейсхолдер:

```text
[[DOCX_IMAGE_img_001]]
```

Формат должен быть стабильным во всех модулях.

---

## 17. Детализация reinsertion после Pandoc

В v1 повторная вставка выполняется **после** `convert_markdown_to_docx_bytes(...)`,
а не во время генерации Markdown.

### 17.1. Почему именно post-processing

Это решение выбрано потому, что:

- текущий pipeline уже стабильно собирает `.docx` через Pandoc;
- не нужно переводить изображения в markdown-ссылки и управлять временными файлами;
- reinsertion можно изолировать в одном месте;
- fallback на оригинал проще реализовать на уровне `python-docx`.

### 17.2. Контракт reinsertion-функции

Функция `reinsert_images_into_docx(...)` должна:

1. принять `docx_bytes` после Pandoc;
2. открыть документ через `Document(...)`;
3. найти все плейсхолдеры `[[DOCX_IMAGE_*]]` в основном потоке;
4. для каждого найденного маркера:
   - выбрать `enhanced_bytes`, если они есть и валидны;
   - иначе выбрать `original_bytes`;
5. вставить изображение максимально близко к позиции плейсхолдера;
6. удалить сам плейсхолдер из итогового пользовательского текста;
7. пересохранить документ в `bytes`.

### 17.3. Упрощающее правило v1

Если плейсхолдер находится:

- в отдельном абзаце — абзац целиком заменяется картинкой;
- внутри текстового абзаца — разрешается v1-упрощение: разбить этот участок на
  соседние runs/абзацы и вставить картинку рядом, сохранив смысловую близость.

Требование v1 — не пиксельная идентичность layout, а сохранение порядка и читаемости.

### 17.4. Что делать, если плейсхолдер не найден

Если конкретный `ImageAsset` не найден в финальном `.docx`:

- это не должно падать как фатальная ошибка всего pipeline;
- событие должно логироваться отдельно;
- итоговый документ все равно должен быть собран.

---

## 18. Детализация шага улучшения изображения

В первой версии image-enhancement должен быть реализован как **изолированный этап**,
который можно отключить без влияния на текстовый pipeline.

### 18.1. Контракт функции

Рекомендуемый контракт:

```python
def enhance_image_asset(image: ImageAsset) -> ImageAsset:
    """
    Возвращает новый или обновленный ImageAsset.
    Никогда не должен ломать обработку всего документа.
    При невозможности улучшения возвращает asset с original_bytes.
    """
```

### 18.2. Правило безопасности

Если невозможно надежно доказать, что улучшение безопасно,
функция должна вернуть исходный файл без генеративных изменений.

### 18.3. Очередность внедрения

В рамках v1 допускается реализация в два шага:

1. сначала `enhance_image_asset(...)` как no-op / pass-through с корректным fallback;
2. затем подключение безопасного enhancement-режима.

Это позволяет сначала стабилизировать сохранение изображений, а потом улучшать качество.

### 18.4. Критерии безопасного enhancement

Если режим улучшения включен, он:

- не меняет композицию;
- не меняет текст внутри изображения;
- не добавляет и не удаляет объекты;
- не меняет данные диаграмм, графиков, схем;
- не преобразует смысл изображения.

## 19. Изменение системного prompt для плейсхолдеров

Текущий `prompts/system_prompt.txt` уже запрещает включать контекст в ответ
и игнорировать инъекции из литературного текста. Для v1 нужно явно добавить
еще одно правило про служебные плейсхолдеры.

Рекомендуемый фрагмент спецификации для prompt:

```text
В целевом блоке могут встречаться служебные плейсхолдеры вида [[DOCX_IMAGE_img_001]].
Это структурные маркеры документа, а не пользовательский текст.
Вы обязаны:
- сохранять их без изменений;
- не удалять их;
- не переименовывать их;
- не перемещать их без крайней необходимости;
- возвращать их в итоговом тексте в том же виде.
```

Это правило должно быть добавлено в системный prompt именно как обязательное,
а не как рекомендация.

## 20. Ошибки, fallback и логирование

Для реализации v1 принимаются следующие правила:

| Сценарий | Действие |
|---|---|
| В документе нет изображений | Pipeline работает как сейчас, без специальных шагов |
| Изображение найдено, но формат не удалось корректно обработать | Asset помечается как `unsupported`, документ продолжает обрабатываться |
| Улучшение изображения завершилось ошибкой | Используется `original_bytes`, статус переводится в `failed` или `original` |
| Pandoc сохранил текст, но конкретный плейсхолдер не найден | Логируется предупреждение, сборка документа не падает |
| Reinsertion конкретной картинки не удался | Ошибка локализуется на уровне этого asset, остальные изображения продолжают обрабатываться |

Главный принцип:

> Проблемы с изображениями не должны ломать литературную обработку текста.

### 20.1. Что нужно логировать

Для каждого `ImageAsset` желательно логировать:

- `image_id`;
- итоговый статус;
- был ли найден плейсхолдер в финальном `.docx`;
- использована ли улучшенная или оригинальная версия;
- краткую причину fallback при ошибке.

## 21. Тестовые сценарии для реализации

Текущая спецификация описывает доработку; при реализации должны быть добавлены
точечные тесты именно на новые контракты.

### 21.1. Unit-тесты

1. Извлечение DOCX без картинок:
   - возвращаются обычные `ParagraphUnit`;
   - список `ImageAsset` пуст.

2. Извлечение DOCX с одной картинкой между абзацами:
   - создается один `ImageAsset`;
   - в тексте появляется правильный плейсхолдер.

3. Извлечение DOCX с текстом и картинкой в одном абзаце:
   - порядок текста и плейсхолдера сохраняется.

4. Fallback при ошибке enhancement:
   - функция не выбрасывает фатальную ошибку наружу;
   - используется оригинальная версия.

5. Reinsertion удаляет плейсхолдер из финального текста:
   - в итоговом `.docx` не остается `[[DOCX_IMAGE_*]]`.

### 21.2. Интеграционные тесты

1. DOCX с 2–3 изображениями проходит полный pipeline:
   - текст обрабатывается;
   - итоговый `.docx` собирается;
   - изображения присутствуют в корректном порядке.

2. DOCX, где одно изображение "ломается" на enhancement-шаге:
   - итоговый документ все равно создается;
   - проблемная картинка берется в оригинале.

### 21.3. Специальная проверка на устойчивость плейсхолдера

Отдельно должна быть проверена гипотеза, что строка:

```text
[[DOCX_IMAGE_img_001]]
```

проходит через:

- текстовую редактуру моделью;
- сборку Markdown;
- конвертацию через Pandoc;

без разрушения формата, мешающего последующему поиску.

Если окажется, что Pandoc или модель системно искажает такой формат,
в реализации v1 должен быть выбран другой устойчивый маркер,
но с сохранением всей описанной архитектуры.

---

## 22. Итоговая реализационная декомпозиция

Для фактической разработки порядок работ должен быть таким:

1. добавить `ImageAsset` в `models.py`;
2. добавить извлечение текста + изображений в `document.py`;
3. убедиться, что placeholder-формат устойчив на тестовом round-trip;
4. добавить reinsertion в `generation.py`;
5. подключить orchestration в `app.py`;
6. только после этого включать enhancement-шаг;
7. затем добавить targeted tests на fallback и корректный порядок изображений.

Такой порядок минимизирует риск: сначала сохраняем картинки как есть,
потом аккуратно улучшаем их качество.

## 23. Расширение до v2: image semantic-redraw-mode

### 23.1. Назначение режима

`image semantic-redraw-mode` — это пользовательский режим, в котором система не просто
сохраняет или мягко улучшает изображение, а может **перерисовать его заново**,
если изображение относится к типам, для которых это полезно:

- диаграммы;
- блок-схемы;
- таблицы;
- инфографика;
- mind-map;
- учебные или бизнес-схемы;
- визуальные схемы процессов и взаимосвязей.

Главный принцип v2:

> Разрешено менять оформление, но запрещено менять смысл, структуру и логические связи изображения.

### 23.2. Отличие от image safe-mode

`image safe-mode`:

- не меняет композицию;
- не перерисовывает заново схему;
- ориентирован на сохранение и мягкое улучшение качества.

`image semantic-redraw-mode`:

- может изменить визуальное оформление;
- может заново нарисовать блоки, стрелки, подписи и компоновку;
- но обязан сохранить исходную семантическую структуру;
- при сомнении обязан откатиться в safe-mode или к оригиналу.

---

## 24. Пользовательский сценарий v2

1. Пользователь загружает `.docx`, содержащий текст и изображения.
2. В sidebar выбирает `image semantic-redraw-mode`.
3. Текстовый pipeline работает как раньше.
4. Для каждого изображения система сначала запускает анализ:
   - что это за тип изображения;
   - есть ли в нем текст;
   - допустима ли семантическая перерисовка;
   - какой prompt-профиль и какой render strategy подходят лучше всего.
5. Если изображение подходит для semantic redraw:
   - система строит новую версию изображения через подходящий pipeline.
6. Если semantic redraw небезопасен или неуверен:
   - система использует safe-mode или оригинал.
7. После этого итоговый `.docx` собирается как и в v1,
   а плейсхолдеры заменяются на redrawn/safe/original изображения.
8. Пользователь получает итоговый документ, где текст литературно отредактирован,
   а схемы и диаграммы при необходимости стали более чистыми и читабельными.

## 25. Изменения UI для v2

### 25.1. Новый пользовательский выбор режима

В `ui.render_sidebar()` должен появиться новый элемент управления:

- `image safe-mode`
- `image semantic-redraw-mode`

Рекомендуемый UX-вариант:

- radio или selectbox в sidebar;

Если выбран `image semantic-redraw-mode`, интерфейс должен явно сообщать:

- что режим экспериментальнее safe-mode;
- что prompt для redraw выбирается автоматически после анализа изображения;
- что для неподходящих изображений система может автоматически откатиться
  к safe-mode или оригиналу.

### 25.3. Что желательно показать в результатах

Для прозрачности обработки желательно добавить в run-log или status-summary:

- `image_id`;
- detected image type;
- выбранный prompt key;
- выбранную render strategy;
- итоговый путь обработки:
  - `semantic_redraw_structured`
  - `semantic_redraw_direct`
  - `safe_mode_fallback`
  - `original_fallback`

Это особенно важно, потому что v2 вводит интеллектуальный выбор prompt’а,
и пользователь должен понимать, почему разные картинки обрабатываются по-разному.

## 26. Архитектура v2

### 26.1. Новые логические слои

Для v2 рекомендуется выделить следующие слои:

1. `Image Extraction Layer`
2. `Image Analysis Layer`
3. `Prompt Registry Layer`
4. `Image Generation Layer`
5. `Validation & Fallback Layer`
6. `DOCX Reinsertion Layer`

### 26.2. Рекомендуемое распределение по модулям

#### `ui.py`

- добавляет выбор image-mode в sidebar;
- показывает пользователю активный режим и предупреждение для semantic redraw.

#### `app.py`

- получает выбор пользователя из UI;
- передает его в pipeline;
- оркестрирует ветвление:
  - `image safe-mode`
  - `image semantic-redraw-mode`

#### `document.py`

- продолжает отвечать за извлечение текста и изображений;
- не должен содержать логики выбора prompt’ов или генерации изображений.

#### `image_analysis.py`

Новый модуль, который:

- анализирует изображение vision-моделью;
- классифицирует тип изображения;
- оценивает, допустим ли semantic redraw;
- выбирает prompt key;
- выбирает render strategy.

#### `image_prompts.py` / `config.py`

Новый или расширенный слой конфигурации, который:

- хранит список prompt’ов по умолчанию для разных типов изображений;
- отдает prompt-template по `prompt_key`;
- отделяет prompt-контент от orchestration-кода.

#### `image_generation.py`

Новый модуль, который:

- выполняет safe enhancement;
- выполняет direct semantic redraw через `gpt-image-1`;
- выполняет structured redraw через JSON/Mermaid/SVG pipeline;
- возвращает валидный `ImageAsset` с итоговым статусом.

### 26.3. Почему v2 лучше выносить в отдельные image-модули

Для v1 допустимо держать image-логику ближе к `generation.py`,
но для v2 это уже становится слишком сложным, потому что нужно отдельно:

- анализировать изображение;
- выбирать prompt;
- выбирать модель;
- выбирать стратегию генерации;
- валидировать результат;
- делать fallback.

Поэтому спецификация v2 рекомендует **не смешивать**
semantic redraw-логику с текстовой генерацией блоков.

## 27. Pipeline v2

```text
Исходный DOCX
    -> Извлечение текста, изображений и плейсхолдеров
    -> Текстовый pipeline редактирования (как в v1)
    -> Для каждого изображения:
         -> Vision-анализ изображения
         -> Классификация типа изображения
         -> Выбор prompt key
         -> Выбор render strategy
         -> Генерация изображения
         -> Валидация результата
         -> Fallback при необходимости
    -> Pandoc -> DOCX
    -> Reinsertion изображений по плейсхолдерам
    -> Готовый DOCX
```

### 27.1. Допустимые стратегии генерации

В v2 допускаются три стратегии:

1. `safe_mode`
   - мягкое улучшение без семантической перерисовки;
2. `semantic_redraw_direct`
   - image-to-image генерация через `gpt-image-1`;
3. `semantic_redraw_structured`
   - анализ структуры -> JSON/Mermaid/Graphviz -> SVG/изображение.

### 27.2. Правило выбора стратегии

По умолчанию для v2 используется следующий приоритет:

1. если изображение хорошо структурируется и важно сохранить точные связи —
   предпочитать `semantic_redraw_structured`;
2. если изображение сложнее, но все еще относится к инфографике/схеме —
   разрешать `semantic_redraw_direct` через `gpt-image-1`;
3. если тип сомнительный или confidence низкий —
   откатываться к `safe_mode`;
4. если и safe-обработка не удалась —
   возвращать оригинал.

## 28. Основные модели и API для v2

### 28.1. Основная модель для semantic redraw

Для **semantic redraw** основной моделью считается
`gpt-image-1` через **OpenAI Images API**.

Причины выбора:

- модель умеет работать по схеме image-to-image;
- хорошо подходит для инфографики, диаграмм и схем;
- умеет перерисовывать изображение в более чистом и современном стиле;
- позволяет использовать исходную картинку как reference.

### 28.2. Модель для анализа изображения

Для анализа изображения и выбора prompt’а рекомендуется использовать
vision-capable text model:

- `gpt-4.1`
- или `gpt-4o`

Именно этот слой должен:

- определить, что изображено;
- понять тип структуры;
- извлечь ключевые подписи и связи;
- оценить, стоит ли идти в `semantic_redraw_direct`
  или в `semantic_redraw_structured`.

### 28.3. Предпочтительный structured pipeline для "строгих" схем

Если требуется максимально точное сохранение структуры,
предпочтительным считается pipeline:

```text
Vision model (gpt-4.1 / gpt-4o)
    -> structured JSON / Mermaid / Graphviz
    -> SVG render
    -> rasterization if needed
```

Это стабильнее, чем чистая image generation,
когда нужно почти детерминированно сохранить:

- число колонок;
- стрелки;
- связи;
- подписи;
- порядок блоков.

---

## 29. Интеллектуальный выбор prompt’а

### 29.1. Общий принцип

При выборе `image semantic-redraw-mode` **не должен использоваться один универсальный prompt**
для всех картинок. Вместо этого должен быть реализован:

- отдельный **default image-analysis prompt**, который активируется самим выбором режима;
- набор prompt’ов по умолчанию для разных типов изображений;
- агент выбора prompt’а после анализа изображения.

То есть последовательность должна быть такой:

1. пользователь выбирает `image semantic-redraw-mode`;
2. pipeline переходит на **другой prompt по умолчанию** — prompt анализа изображения;
3. analysis-agent определяет тип изображения и стратегию;
4. затем выбирается конкретный redraw-prompt из registry по `prompt_key`.

### 29.2. Контракт анализа

Результатом image-analysis шага должен быть объект вида:

```python
@dataclass
class ImageAnalysisResult:
    image_type: str
    contains_text: bool
    semantic_redraw_allowed: bool
    confidence: float
    prompt_key: str
    render_strategy: str
    structure_summary: str
    extracted_labels: list[str]
    fallback_reason: str | None = None
```

### 29.3. Задача analysis-agent

Этот агент должен ответить на вопросы:

1. Что это за тип изображения?
2. Это фото, скриншот, диаграмма, таблица, mind-map, инфографика или смешанный тип?
3. Нужно ли сохранять строгую структуру?
4. Можно ли безопасно выполнить semantic redraw?
5. Какой prompt key подходит лучше всего?
6. Какую стратегию рендеринга выбрать?

### 29.4. Правило confidence

Если confidence анализа недостаточен, система не должна пытаться "угадать"
сильную перерисовку. В таком случае применяется:

- `safe_mode`
- или оригинал.

## 30. Реестр prompt’ов по умолчанию

### 30.1. Требование к реестру

Список prompt’ов по умолчанию должен быть вынесен из основного orchestration-кода
и храниться как отдельный конфигурируемый registry.

У каждого prompt-профиля должны быть:

- `prompt_key`
- список подходящих типов изображений
- рекомендуемая стратегия
- шаблон инструкции для модели
- ограничения на изменение смысла

### 30.2. Минимальный набор prompt-профилей v2

| `prompt_key` | Для каких изображений | Предпочтительная стратегия | Назначение |
|---|---|---|---|
| `diagram_semantic_redraw` | блок-схемы, процессные схемы | `semantic_redraw_structured` | сохранить блоки и стрелки, улучшить оформление |
| `table_semantic_redraw` | таблицы, табличные схемы | `semantic_redraw_structured` | сохранить строки/колонки и подписи |
| `infographic_semantic_redraw` | инфографика, многоуровневые схемы | `semantic_redraw_direct` | сохранить смысл, улучшить визуальную подачу |
| `mindmap_semantic_redraw` | mind-map, concept map | `semantic_redraw_structured` | сохранить центральные и дочерние узлы |
| `chart_semantic_redraw` | графики и диаграммы | `semantic_redraw_structured` или `safe_mode` | очень осторожный redraw только при высокой уверенности |
| `screenshot_safe_fallback` | скриншоты интерфейса | `safe_mode` | по умолчанию не перерисовывать как схему |
| `photo_safe_fallback` | фотографии | `safe_mode` | semantic redraw не является режимом по умолчанию |

### 30.3. Пример prompt для диаграммы

Ниже приведен эталонный пример prompt-профиля,
который может использоваться для `diagram_semantic_redraw`:

```text
Redraw this diagram as a clean modern vector infographic.

Keep the same semantic structure and relationships.
Preserve the same number of columns, blocks, arrows, and labels.
Do not invent new entities and do not remove existing ones.
If text is readable, preserve the wording as accurately as possible.

Style:
clean academic infographic,
vector diagram,
aligned boxes,
clear arrows,
white background,
modern typography.
```

### 30.4. Обязательные ограничения для всех semantic-redraw prompt’ов

Каждый prompt такого типа должен явно запрещать:

- менять число сущностей без основания;
- менять направление стрелок;
- удалять подписи;
- заменять значения, цифры и категории;
- добавлять новые смысловые объекты;
- менять логику связей.

## 31. Правила выбора между direct redraw и structured redraw

### 31.1. Когда выбирать structured redraw

`semantic_redraw_structured` должен выбираться в первую очередь, если:

- изображение похоже на таблицу;
- изображение похоже на блок-схему;
- важны точные стрелки и связи;
- важна высокая воспроизводимость результата;
- есть шанс выразить структуру в JSON/Mermaid/Graphviz.

### 31.2. Когда выбирать direct redraw

`semantic_redraw_direct` через `gpt-image-1` допустим, если:

- изображение является инфографикой или схемой со сложным оформлением;
- структура понятна, но не слишком удобно сводится к Mermaid/Graphviz;
- важна эстетическая очистка и современная подача;
- строгая детерминированность не обязательна.

### 31.3. Когда не выбирать semantic redraw

Semantic redraw не должен включаться автоматически даже в v2, если:

- это фотография;
- это скриншот интерфейса;
- это документ-скан;
- на изображении слишком много мелкого текста, который нельзя надежно восстановить;
- агент анализа не уверен в структуре.

В таких случаях система должна использовать:

- `image safe-mode`
- либо оригинал.

## 32. Изменения моделей данных и конфигурации

### 32.1. Расширение `ImageAsset`

Для v2 `ImageAsset` желательно расширить полями:

- `mode_requested` — `safe` / `semantic_redraw`;
- `analysis_result` — ссылка на `ImageAnalysisResult` или сериализуемый dict;
- `prompt_key`;
- `render_strategy`;
- `redrawn_bytes`;
- `final_variant` — `original` / `safe` / `redrawn`;
- `final_reason`.

### 32.2. Новые конфигурационные сущности

На уровне конфигурации нужны:

- default image mode: `safe`;
- default semantic-redraw model: `gpt-image-1`;
- default image-analysis model: `gpt-4.1` или `gpt-4o`;
- registry default prompts;
- порог confidence для перехода в semantic redraw;
- флаг `prefer_structured_redraw`.

### 32.3. Изменение prompt-loading слоя

Текущая загрузка одного `system_prompt.txt` недостаточна для image-v2.
Нужны как минимум:

- текстовый system prompt для литературной редактуры;
- отдельный default prompt для `image semantic-redraw-mode`, то есть prompt для image-analysis;
- registry prompt’ов для типов изображений;
- optional validator prompt для проверки результата redraw.

При этом важно явно разделить:

- `text system prompt` — для литературной обработки текстовых блоков;
- `image analysis prompt` — default prompt для режима semantic redraw;
- `image redraw prompt templates` — набор специализированных prompt’ов,
  которые выбираются после анализа изображения.

## 33. Безопасность и fallback в v2

### 33.1. Главный принцип v2

> Лучше вернуть safe-mode или оригинал, чем красиво перерисовать схему с потерей смысла.

### 33.2. Обязательные fallback-правила

1. Если analysis-agent не уверен в типе изображения — использовать `safe_mode`.
2. Если prompt key не может быть выбран надежно — использовать `safe_mode`.
3. Если structured redraw не дал валидную структуру — пробовать direct redraw только если это допустимо.
4. Если direct redraw визуально подозрителен или теряет элементы — использовать `safe_mode`.
5. Если safe-mode тоже недоступен — использовать оригинал.
6. Ошибка на одном изображении не должна ломать сборку всего документа.

### 33.3. Что считается подозрительным результатом

Результат semantic redraw считается подозрительным, если:

- исчезли блоки;
- исчезли стрелки;
- изменилось число колонок;
- пропали подписи;
- появились новые сущности;
- цифры/значения перестали совпадать;
- текст стал явно нечитаемым или бессмысленным.

## 34. Критерии приемки v2

Решение считается соответствующим v2, если:

1. пользователь может явно выбрать `image semantic-redraw-mode` в интерфейсе;
2. для этого режима используется другой базовый image-pipeline, чем в safe-mode;
3. по умолчанию существует **несколько** prompt-профилей для разных типов изображений;
4. prompt-профиль выбирается автоматически после анализа изображения;
5. для строгих схем допускается structured redraw pipeline;
6. для image-to-image semantic redraw используется `gpt-image-1`;
7. при низкой уверенности система откатывается к safe-mode или оригиналу;
8. текстовый pipeline редактирования документа остается совместимым с v1;
9. итоговый `.docx` по-прежнему собирается даже при частичных сбоях на image-v2 этапе.

## 36. Итоговое проектное решение

Для проекта рекомендуется считать:

- v1 = `image safe-mode`
- v2 = `image semantic-redraw-mode`

Причем v2 должен строиться **поверх** v1, а не вместо него:

- safe-mode остается базовым и дефолтным;
- semantic-redraw-mode является отдельной пользовательской опцией;
- выбор prompt’а выполняется не статически, а интеллектуально после image-analysis;
- `gpt-image-1` используется как основная модель direct semantic redraw;
- для строгих диаграмм и таблиц предпочтителен structured JSON/Mermaid/SVG pipeline.

Именно такая схема дает лучший баланс между:

- качеством визуальной перерисовки;
- сохранением смысла;
- управляемостью результата;
- совместимостью с уже существующей архитектурой приложения.
