# UI Live Panel — Декомпозиция и рефакторинг

**Дата:** 2026-03-16
**Статус:** План для согласования перед реализацией
**Контекст:** Два стойко воспроизводимых бага в live processing panel + архитектурные проблемы

---

## 1. Зафиксированные баги

### BUG-1: Лента «События» не скроллится к последней строке

**Симптом:** При обработке документа контейнер `activity-feed-items` не прокручивается к последнему элементу. Пользователь вынужден скроллить вручную.

**Корневая причина:**

`_scroll_activity_feed_to_latest()` инжектирует JavaScript через `streamlit.components.v1.html()`, который рендерится в **изолированный iframe** (height=0, width=0). Скрипт пытается найти feed-элемент в родительском DOM через `window.parent.document.getElementById(...)`.

Проблема: Streamlit добавляет в iframe атрибут `sandbox`, который блокирует доступ к `parent.document`. Даже если same-origin политика позволяет доступ, Streamlit-фрагменты пересоздают DOM-поддерево каждую секунду, и к моменту исполнения таймеров (50–1000 мс) feed-элемент может быть уже заменён новым.

Дополнительно: `MutationObserver` наблюдает за `observerTarget`, который может быть `window.document.body` iframe'а (пустой), а не основного документа.

**Вывод:** Вся стратегия «JS в отдельном iframe ищет элемент в родительском DOM» ненадёжна в Streamlit.

---

### BUG-2: «Текущий Markdown» показывает placeholder-контент и мерцает

**Симптом:** Во время обработки виджет показывает блок `[[DOCX_IMAGE_img_007]]` (placeholder без реального контента). Expander и selectbox визуально пересоздаются каждую секунду.

**Корневые причины (четыре):**

1. **Блоки-заглушки попадают в selectbox.** `processed_block_markdowns` содержит ВСЕ блоки, включая те, которые состоят только из `[[DOCX_IMAGE_img_XXX]]`. Selectbox показывает номера 1..N без фильтрации — пользователь может выбрать (или auto-follow выберет) блок-заглушку.

2. **`render_partial_result()` не передаёт `focus_latest=True`.** Во время активной обработки вызывается:
   ```python
   render_markdown_preview(
       title="Текущий Markdown",
       preferred_block_index=preview_block_index,
   )
   ```
   Параметр `preferred_block_index` используется только при начальной инициализации `select_state_key`. После первого рендера `select_state_key` уже установлен, и `preferred_block_index` игнорируется — виджет показывает то, что пользователь выбрал (или что попало в state ранее).

3. **Мерцание из-за фрагмента.** `render_processing_panel` — это `@st.fragment(run_every=1)`. Каждый ререндер пересоздаёт все дочерние виджеты. Хотя `widget_key` стабилен при неизменном `option_count`, сам expander + selectbox + text_area пересоздаются, вызывая визуальное мерцание.

4. **Сложная система ключей.** Пять ключей (`legacy_select_key`, `select_state_key`, `widget_key`, `follow_latest_key`, `last_count_key`) для одного selectbox. Код трудно читать, отлаживать и тестировать. Поведение зависит от порядка вызовов и стейт-миграции между ключами.

## 2. Архитектурные проблемы

### ARCH-1: JS-скроллинг через iframe

Steamlit `components.v1.html()` предназначен для встраивания пользовательских компонентов, а не для манипуляции основным DOM приложения. Попытка найти и скроллить элемент родительского документа из iframe — ненадёжный анти-паттерн.

### ARCH-2: Monolithic fragment

`render_processing_panel` перерисовывает ВСЁ каждую секунду: status card + activity feed + run log + image validation + partial result + controls. Это:
- тяжело для рендеринга (лишние перерисовки стабильных элементов);
- вызывает мерцание виджетов (selectbox, expander);
- увеличивает риск конфликтов ключей и стейт-гонок.

### ARCH-3: Отсутствие фильтрации блоков

`render_markdown_preview` оперирует «сырым» списком `processed_block_markdowns` без понимания семантики блоков. Блоки-placeholder'ы (только `[[DOCX_IMAGE_img_XXX]]`), пустые блоки и блоки в процессе обработки не отличаются от блоков с реальным контентом.

### ARCH-4: Две роли одного виджета

`render_markdown_preview` используется для двух разных задач:
1. **Live preview** во время обработки (через `render_partial_result`) — тут нужен auto-follow к последнему готовому блоку.
2. **Final preview** после завершения (через `render_result_bundle`) — тут нужен свободный выбор блока.

Эти две роли имеют разные UX-требования, но реализованы одним виджетом с флагами `focus_latest` / `preferred_block_index`, что порождает хрупкую логику.

## 3. Предлагаемый план рефакторинга

### Фаза 1: Исправить скроллинг ленты событий

**Задача 1.1:** Заменить iframe-JS на CSS-only auto-scroll.

Вместо `_scroll_activity_feed_to_latest()` с iframe-JS использовать CSS-трюк:

```css
.activity-feed-items {
    display: flex;
    flex-direction: column-reverse;
    /* ... existing styles ... */
}
```

Идея: развернуть flex-direction, чтобы контейнер рендерил элементы снизу вверх. Последний элемент (визуально внизу) будет всегда visible. НО это разворачивает порядок элементов — нужно инвертировать render-порядок в Python.

Альтернативный вариант: добавить якорный `<div>` в конец списка и использовать CSS `scroll-snap` или `anchor` для автоскролла без JS.

**Задача 1.2:** Если CSS-only недостаточно — Inline JS через `st.markdown`.

Перенести скролл-скрипт из `components.v1.html()` в `st.markdown(..., unsafe_allow_html=True)`. В этом случае `<script>` выполнится прямо в основном DOM Streamlit-страницы, и `document.getElementById()` точно найдёт элемент.

**Проверка:** Нужен E2E-тест (или ручной тест), подтверждающий, что последний элемент виден после добавления новых записей.

**Задача 1.3:** Удалить `_scroll_activity_feed_to_latest()` и все связанные с ней retry/MutationObserver конструкции.

---

### Фаза 2: Исправить контент Markdown-превью

**Задача 2.1:** Фильтровать блоки-заглушки.

В `render_markdown_preview` (или в новой обёртке `render_live_markdown_preview`):
- Не показывать в selectbox блоки, состоящие только из `[[DOCX_IMAGE_img_XXX]]` или пустые.
- Показывать только содержательные блоки (с реальным текстом).
- Если ни одного содержательного блока нет — скрывать виджет целиком.

Реализация: создать `_meaningful_blocks(blocks) -> list[tuple[int, str]]`, который фильтрует блоки с помощью `_strip_docx_image_placeholders()`.

**Задача 2.2:** Передавать `focus_latest=True` из `render_partial_result`.

Во время обработки Markdown-превью должно автоматически следовать за последним содержательным блоком. Только ручной выбор пользователем другого блока должен отключать auto-follow.

```python
def render_partial_result() -> None:
    # ...
    render_markdown_preview(
        title="Текущий Markdown",
        focus_latest=True,  # ← auto-follow последний содержательный
    )
```

**Задача 2.3:** Убрать `preferred_block_index` — он всегда компенсировал отсутствие `focus_latest=True`.

После включения `focus_latest=True` в `render_partial_result()`, параметр `preferred_block_index` больше не нужен — его логика полностью поглощается `focus_latest` + автоматической фильтрацией заглушек.

---

### Фаза 3: Упростить state-management preview

**Задача 3.1:** Сократить ключи до двух.

Целевое состояние:
- `mdpreview_{hash}_selected` — номер выбранного блока (non-widget state).
- `mdpreview_{hash}_count` — последнее известное количество блоков (для определения new_block_arrived).

Удалить:
- `legacy_select_key` — миграция больше не нужна (одноразовая, уже отработала).
- `widget_key` с `option_count` в имени — использовать стабильный ключ, не привязанный к количеству опций.
- `follow_latest_key` — определять auto-follow состояние из сравнения `selected == count`.

**Задача 3.2:** Рассмотреть отказ от selectbox в live-превью.

Во время обработки selectbox для выбора блока не особо полезен: пользователь хочет видеть последний обработанный блок, а не переключаться между ними. Вместо selectbox можно показывать:
- Просто последний содержательный блок (без выбора).
- Заголовок `Блок N / M` + содержимое.

Selectbox оставить только для `render_result_bundle` (post-processing preview).

---

### Фаза 4: Уменьшить мерцание

**Задача 4.1:** Вынести partial result из фрагмента.

`render_partial_result` — тяжёлая функция (expander + selectbox + text_area). Перерендер каждую секунду вызывает мерцание. Варианты:
- Рендерить Markdown-превью вне `@st.fragment(run_every=1)`, обновляя его только по событию (через `st.rerun()` при new_block_arrived).
- Оставить preview внутри fragment, но рендерить **только text** (без selectbox/expander) как `st.code(block_text, language="markdown")`.

**Задача 4.2:** Увеличить `run_every` для rendering panel.

1 секунда — агрессивный интервал. Блоки обрабатываются в среднем 3–10 секунд. Увеличение до 2–3 секунд снизит нагрузку и мерцание вдвое.
