# Стратегия монетизации DocxAICorrector

## 1. Структура затрат на токены

Основные потребители токенов в проекте:

| Операция | Модель | Стоимость за документ ~50 стр. |
|---|---|---|
| Текстовая правка блоков | gpt-5-mini | ~$0.15–0.40 |
| Image analysis (vision) | gpt-4.1 | ~$0.05–0.15 за изображение |
| Image validation (vision) | gpt-4.1 | ~$0.05–0.10 за изображение |
| Semantic redraw (генерация) | DALL-E / модель | ~$0.04–0.08 за попытку |
| Compare-all (3 варианта) | всё вместе | ×3 от одиночного |

**Документ на 50 страниц с 10 изображениями в режиме `compare_all` может стоить $2–5 в себестоимости.**

## 2. Как не вылететь в трубу

### А. Жёсткие лимиты на стороне сервера
- Лимит страниц или символов на документ по тарифу
- Лимит изображений на документ
- Бюджет модельных вызовов (`semantic_redraw_max_model_calls_per_image` уже есть — расширить на весь пайплайн)
- Дефолтный режим `safe` для изображений (дешевле всего), `semantic_redraw` и `compare_all` — только на старших тарифах

### Б. Выбор моделей по тарифу

| Тариф | Текстовая модель | Image pipeline |
|---|---|---|
| Free / Starter | gpt-5-mini | safe only |
| Pro | gpt-5-mini / gpt-5.2 | semantic_redraw_direct |
| Business | gpt-5.4 / gpt-5.4-pro | compare_all |

### В. Кеширование и дедупликация
- Кешировать анализ изображений — если тот же файл загружен повторно, не пересчитывать
- Кешировать system prompt через OpenAI prompt caching (50% скидка на кешированные токены)
- При рестарте (`restart_store.py`) не повторять уже обработанные блоки

### Г. Предварительная оценка стоимости
- Перед запуском — показать пользователю оценку: «Документ содержит X блоков, Y изображений. Расход: ~Z кредитов»
- Пользователь подтверждает перед списанием

## 4. Автоматизация платежей

### Сравнение решений

| Решение | Плюсы | Стоимость |
|---|---|---|
| **Stripe** | Стандарт индустрии, подписки + usage-based billing, webhooks, Customer Portal | 2.9% + $0.30 за транзакцию |
| **Paddle** | Merchant of Record (берёт на себя налоги/НДС), проще для соло | 5% + $0.50 |
| **LemonSqueezy** | MoR, простой UI, хорош для инди | 5% + $0.50 |
| **ЮKassa** (РФ) | Для российских клиентов | ~3.5% |

**Рекомендация: Stripe** — лучшая поддержка usage-based billing, что критично для кредитной модели.

### Архитектура интеграции

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  Streamlit   │────▶│  FastAPI      │────▶│  Stripe API  │
│  (фронтенд)  │     │  (бэкенд)     │     │  (платежи)   │
└─────────────┘     └──────────────┘     └─────────────┘
                           │                      │
                           ▼                      ▼
                    ┌──────────────┐     ┌─────────────┐
                    │  PostgreSQL   │     │  Webhooks    │
                    │  (users,      │◀────│  (подписки,  │
                    │   credits,    │     │   оплата)    │
                    │   usage)      │     └─────────────┘
                    └──────────────┘
```

### Ключевые шаги реализации

1. **Auth слой** — добавить авторизацию (Streamlit Authenticator или вынести фронтенд на Next.js + Streamlit как backend worker)
2. **Модель пользователя** — PostgreSQL: `users`, `subscriptions`, `credit_balance`, `usage_log`
3. **Middleware проверки лимитов** — перед каждым вызовом OpenAI проверять баланс/лимит
4. **Stripe integration**:
   - `stripe.checkout.Session` — для подписок
   - `stripe.billing.Meter` — для usage-based биллинга (кредиты)
   - Webhooks: `invoice.paid`, `customer.subscription.updated`, `customer.subscription.deleted`
5. **Трекинг расхода** — логировать каждый API-вызов с `usage.prompt_tokens` + `usage.completion_tokens` из ответа OpenAI, списывать кредиты

## 5. Быстрый путь к MVP монетизации

Минимальный набор для запуска:

1. **LemonSqueezy или Stripe Checkout** — кнопка оплаты, без своего бэкенда для платежей
2. **Лицензионный ключ** — после оплаты выдаётся ключ, вводится в Streamlit
3. **SQLite + файл лимитов** — хранить usage локально, проверять при старте
4. **Cloudflare Tunnel или Railway** — деплой Streamlit с HTTPS

Это можно запустить за пару дней, а потом итерировать к полноценному SaaS.
