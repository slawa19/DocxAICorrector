# Copilot CLI loop: короткая инструкция

Эта инструкция описывает самый простой способ запускать автономную разработку из KiloCode UI без `Copilot Orchestrator Bridge`.

## Короткий ответ

Используйте команду:

```text
/copilot-loop <описание задачи>
```

KiloCode может сделать до 3 итераций:

```text
Copilot CLI edits -> git diff -> targeted checks -> next iteration or final result
```

Плагин bridge для этого не нужен. Он был text-only транспортом и не должен быть основным development runtime.

`До 3 итераций` означает верхний предел, а не обязательный план. Если задача решена после первой итерации, loop должен завершиться сразу.

Toolkit scripts лежат в `tools/copilot-cli/`. Базовая smoke-проверка runtime:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\copilot-cli\scripts\smoke-copilot-edit.ps1
```

## Как формулировать задачу

Лучший шаблон:

```text
/copilot-loop
Цель: <какое пользовательское или техническое поведение нужно получить>
Контекст: <где проблема видна, какие файлы/экраны/тесты связаны>
Границы: <что нельзя менять>
Проверка: <как понять, что задача готова>
Автономность: сделай до 3 итераций, не коммить, спрашивай только при важной развилке.
```

## Как трактовать лимит итераций

`tools/copilot-cli/README.md` использует `до 3 итераций` как рекомендуемый дефолт для общего development loop.

Это не жесткое требование toolkit и не план обязательно сделать несколько шагов. Если задача узкая или вы хотите более короткий автономный цикл, допустимо явно задать меньший лимит, например `до 2 итераций`.

Практическое правило:

- `до 3 итераций` - базовый и рекомендуемый режим по умолчанию;
- `до 2 итераций` - нормальный вариант для узких doc-only, test-only или локально ограниченных задач;
- важнее явно указать лимит в prompt, чем всегда использовать одно и то же число.

Минимальный шаблон:

```text
/copilot-loop Исправь <проблему>. Сделай до 3 итераций автономно, не коммить, остановись только при важной развилке или завершении.
```

## Примеры хороших задач

```text
/copilot-loop
Цель: починить bulk selection на странице выбора глав, чтобы кнопки "Выбрать всё" и "Снять всё" реально обновляли чекбоксы.
Контекст: Streamlit UI в src/docxaicorrector/ui/_app.py, историческая спецификация docs/archive/specs/STRUCTURE_REVIEW_PANEL_UX_REFACTOR_SPEC_2026-05-08.md.
Границы: не менять pipeline перевода и real-document validation.
Проверка: точечные тесты вокруг structure review panel или ближайшие существующие тесты.
Автономность: до 3 итераций, не коммить, спрашивай только при архитектурной развилке.
```

```text
/copilot-loop
Цель: уменьшить диагностический шум в UI после подготовки документа и оставить только список глав и кнопки запуска.
Контекст: реализовать только поведение из docs/archive/specs/STRUCTURE_REVIEW_PANEL_UX_REFACTOR_SPEC_2026-05-08.md.
Границы: без новых цветов, без нового дизайна, без переписывания extraction pipeline.
Проверка: targeted pytest для UI/state helpers; если тестов нет, вернуть это как риск.
Автономность: максимум 3 итерации.
```

```text
/copilot-loop
Цель: добавить regression test для бага, при котором PDF structural preparation ошибочно относит главы к Front Matter.
Контекст: смотри docs/archive/specs/AI_FIRST_STRUCTURE_RECOVERY_SPEC_2026-05-08.md и tests вокруг structural preparation.
Границы: сначала только тест, production code менять только если тест невозможно выразить без минимального helper.
Проверка: запустить конкретный test node через canonical test path.
Автономность: до 2 итераций, не коммить.
```

## Примеры плохих задач

```text
/copilot-loop Сделай проект лучше.
```

```text
/copilot-loop Полностью перепиши UI и pipeline.
```

```text
/copilot-loop Исправь все баги.
```

Такие формулировки слишком широкие. KiloCode не сможет безопасно понять границы, критерий готовности и момент остановки.

## Как получить читаемый результат

Команда `/copilot-loop` должна завершаться отчетом в таком формате:

```text
Результат
- Выполнено: <что реально изменено>
- Изменены файлы: <пути>
- Проверки: <что запускалось и чем закончилось>
- Итерации: <сколько итераций сделано>
- Риски: <что осталось непроверенным или спорным>

Продолжение
<готовый prompt для продолжения в этой или новой KiloCode-сессии>
```

Если задача завершена, блок `Продолжение` не обязателен. Он нужен только если работа реально не завершена, требуется handoff в новую сессию или полезно сохранить точный следующий шаг без пересказа истории.

Если вы хотите продолжить работу, копировать историю не нужно. Используйте блок `Продолжение` целиком или слегка поправьте направление.

## Как минимизировать копипасту

Формулируйте задачу так, чтобы KiloCode сам вел loop:

```text
Верни в финале самодостаточный continuation prompt: текущий статус, измененные файлы, проверки, оставшиеся шаги и ограничения.
```

Для новой сессии достаточно вставить continuation prompt из прошлого результата. Если направление ушло не туда, измените только блок `Следующая цель` или добавьте одно ограничение.

## Когда KiloCode должен спрашивать пользователя

KiloCode не должен спрашивать про мелочи. Он должен остановиться только если:

- есть несколько архитектурно разных путей;
- нужно удалить или массово переписать важный код;
- изменения пересекаются с чужими незавершенными правками;
- нужны секреты, доступы или ручная настройка;
- тесты/диагностика дают неоднозначный результат;
- задача стала шире исходных границ.

## Нужно ли использовать bridge-плагин

Нет, не для development loop.

Bridge-плагин можно удалить или оставить только как legacy/debug artifact. Основной путь должен быть:

```text
KiloCode -> Copilot CLI -> git diff/tests -> KiloCode
```

Toolkit для переноса в новый проект:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\copilot-cli\scripts\install-into-project.ps1 `
  -TargetProjectPath "D:\path\to\another-project"
```
