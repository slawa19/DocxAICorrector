# Спецификация follow-up: quality hardening для image pipeline

## 0. Назначение документа

Этот документ фиксирует **отдельный следующий этап** после закрытия базового Level 1 guardrail
для `image semantic-redraw-mode`.

Его цель — не расширить уже принятый Level 1 core-contract, а описать отдельный пакет доработок,
который улучшает качество эвристик, диагностируемость и устойчивость image pipeline на реальных кейсах.

Этот этап должен рассматриваться как **follow-up hardening / quality iteration**, а не как
незакрытый обязательный минимум предыдущего merge.

---

## 1. Scope follow-up этапа

В этот follow-up этап входят шесть направлений:

1. улучшение качества `image_analysis` эвристик;
2. улучшение качества `image_generation` эвристик;
3. усиление извлечения `text/label signals`;
4. quality tuning для `prompt_key`, `render_strategy`, thresholds;
5. возможное расширение UI / run-log;
6. дальнейший hardening по `false positive / false negative`.

---

## 2. Цели этапа

Этот этап должен:

1. снизить число ошибок классификации изображений перед redraw;
2. повысить качество выбора prompt-профиля и render strategy;
3. уменьшить вероятность semantic drift и потери text/label signals;
4. улучшить наблюдаемость pipeline через UI и run-log;
5. дать команде базу для data-driven tuning по реальным регрессиям и production-кейсам.

---

## 3. Не входит в scope

В этот этап по-прежнему не обязаны входить:

- полноценный OCR-suite с гарантированным восстановлением любого текста на изображении;
- proof-level structural equivalence для всех типов схем и диаграмм;
- полная поддержка всех Word graphic/layout edge cases;
- переписывание текущего Level 1 orchestration flow с нуля;
- требование идеального визуального совпадения для каждого изображения.

---

## 4. Детализация доработок

### 4.1. Улучшение качества `image_analysis` эвристик

Цель — уйти от грубой MIME / general-purpose classification в сторону более содержательного
анализа типа изображения.

Минимально ожидаемые направления:

- отличать diagram / chart / screenshot / scanned-text / photo-like cases надежнее текущего слоя;
- учитывать состав сигнала: текст, стрелки, подписи, таблицеподобные области, графические примитивы;
- уменьшать число случаев, когда semantic redraw запускается для заведомо неподходящих изображений.

### 4.2. Улучшение качества `image_generation` эвристик

Цель — повысить качество candidate generation, не ломая conservative fallback policy.

Минимально ожидаемые направления:

- улучшить выбор между `safe`, `semantic_redraw_direct`, `semantic_redraw_structured`;
- усилить quality gates до принятия финального варианта;
- сохранить совместимость с текущим validator-driven accept/fallback flow.

### 4.3. Усиление извлечения `text/label signals`

Цель — уменьшить потери смысловых подписей и текстовых маркеров на изображениях.

Минимально ожидаемые направления:

- точнее извлекать labels, legend, axis text, callouts, title/subtitle;
- улучшить обнаружение частичной потери текста между original и candidate;
- использовать эти сигналы в validator / decision routing без слома текущего Level 1 flow.

### 4.4. Quality tuning для `prompt_key`, `render_strategy`, thresholds

Цель — сделать routing и fallback более предсказуемыми на реальных данных.

Минимально ожидаемые направления:

- уточнять mapping `analysis -> prompt_key`;
- донастраивать выбор `render_strategy`;
- корректировать conservative thresholds на основе run-log и накопленных регрессий.

### 4.5. Возможное расширение UI / run-log

Цель — дать команде более прозрачную per-image диагностику без перегрузки базового интерфейса.

Минимально ожидаемые направления:

- показывать причину выбора стратегии и fallback;
- добавлять summary по validator signals;
- при необходимости раскрывать per-image trace в UI или log-friendly формате.

### 4.6. Hardening по `false positive / false negative`

Цель — стабилизировать поведение validator-driven решения на реальных документах.

Минимально ожидаемые направления:

- собирать и классифицировать регрессии;
- анализировать причины ложных accept / ложных fallback;
- точечно настраивать validator thresholds и decision heuristics.

---

## 5. Предлагаемый порядок реализации

Рекомендуемый порядок выполнения:

1. сначала улучшить `image_analysis` и извлечение `text/label signals`;
2. затем настроить `prompt_key`, `render_strategy` и thresholds;
3. после этого усиливать `image_generation` heuristics;
4. параллельно накапливать false positive / false negative кейсы;
5. UI / run-log расширять только в объеме, который помогает tuning и regression triage.

Такой порядок снижает риск преждевременного усложнения UI и помогает сначала улучшить
качество decision inputs, а уже потом дорабатывать presentation layer.

---

## 6. Критерии приемки follow-up спецификации

Эта follow-up спецификация считается принятой, если одновременно выполняются все условия:

1. существует **отдельный документ**, посвященный именно оставшимся image follow-up доработкам;
2. в документе перечислены все шесть направлений доработок:
   `image_analysis`, `image_generation`, `text/label signals`, tuning,
   UI / run-log, `false positive / false negative`;
3. для каждого направления дано хотя бы краткое описание цели или ожидаемого результата;
4. в документе явно зафиксировано, что это **следующий этап после Level 1**, а не незакрытый
   core-scope базового guardrail;
5. в документе есть отдельный раздел с критериями приемки;
6. в документе есть отдельный test plan, который можно проверить автоматическим тестом.

---

## 7. Объективная проверка / test plan

Для этой follow-up спецификации достаточно следующей автоматической проверки:

- pytest-тест читает markdown-файл follow-up спецификации;
- тест проверяет, что документ существует как отдельный spec в `docs/`;
- тест проверяет наличие разделов `Scope`, `Цели`, `Не входит в scope`,
  `Критерии приемки`, `Объективная проверка / test plan`;
- тест проверяет наличие всех шести направлений доработок;
- тест проверяет наличие формулировки, что это следующий этап после Level 1;
- тест проверяет, что документ явно описывает objective acceptance boundaries.

Если этот тест проходит, значит отдельная спецификация на оставшиеся доработки,
её acceptance criteria и её testable structure зафиксированы объективно.

---

## 8. Как использовать этот документ

Если команда обсуждает:

- **что уже обязательно должно быть в Level 1** — ориентиром остается основной spec
  `Level 1 post-check для image semantic-redraw-mode`;
- **что нужно делать следующим пакетом улучшений** — ориентиром становится этот follow-up spec;
- **какие доработки можно планировать независимо и итеративно** — ориентиром служат шесть
  направлений из раздела scope.

Таким образом, основной spec отвечает за базовый контракт, а этот документ — за следующий
quality-hardening roadmap.
