# Правила разработки для ИИ-агента в проекте DocxAICorrector

**Назначение:** этот документ фиксирует обязательные правила для ИИ-агента, чтобы не повторять архитектурные, интеграционные и quality-ошибки, выявленные в код-ревью проекта.

**Область действия:** все изменения в `app.py`, `document.py`, `image_analysis.py`, `image_generation.py`, `image_validation.py`, `processing_runtime.py`, `config.py`, `ui.py`, `models.py`, `state.py`, тестах и связанных markdown-спецификациях.

---

## 0. Защищённый контракт тестового workflow

WSL/bash-путь выполнения тестов считается защищённым контрактом репозитория.

ИИ-агенту запрещено:

- возвращать PowerShell-обёртки для запуска тестов;
- запускать тесты через PowerShell pipe bridge или иные промежуточные shell-chain обходы;
- менять `.vscode/tasks.json` test tasks так, чтобы канонический путь перестал быть `bash scripts/test.sh ...` в WSL;
- менять `scripts/test.sh`, `.vscode/tasks.json`, `tests/test_script_workflow_smoke.py` и документы про тестовый workflow в рамках несвязанной задачи.

При выполнении тестов в VS Code ИИ-агент должен использовать видимый для пользователя путь финальной верификации:

- существующие VS Code tasks `Run Full Pytest`, `Run Current Test File`, `Run Current Test Node`, если они соответствуют нужному scope;
- если подходящего VS Code task нет, агент обязан явно сказать об этом, а не подменять финальную верификацию своим terminal capture.

Нельзя считать hidden/background terminal capture финальной верификацией, если затем не был выполнен видимый пользовательский прогон.

Уточнение:

- вывод, который видит только сам агент через tool capture, не считается видимым терминалом VS Code;
- вывод из foreground agent shell тоже не считается видимой пользовательской панелью терминала VS Code;
- если пользователь явно хочет видеть прогон в терминале редактора, финальная верификация должна идти через существующие VS Code tasks, а не через agent-side shell capture или foreground tool terminals;
- лог-файлы, изолированные rerun-прогоны и foreground agent shell допустимы только как диагностический слой, но не как финальное доказательство для пользователя.

Если пользователь явно просит изменить сам test workflow, ИИ-агент обязан в одном change-set:

- обновить `scripts/test.sh`;
- синхронизировать `.vscode/tasks.json`;
- обновить `tests/test_script_workflow_smoke.py`;
- синхронизировать `README.md`, `CONTRIBUTING.md` и `docs/WORKFLOW_AND_IMAGE_MODES.md`;
- прогнать `bash scripts/test.sh tests/test_script_workflow_smoke.py -q` и затем полный `bash scripts/test.sh tests/ -q`.

Смысл правила: тестовый контракт должен ломаться только намеренно, обзорно и с явным обновлением всех уровней source of truth.

## 0.0. Защищённый контракт качества тестов

ИИ-агенту запрещено возвращать в основной regression-suite два класса низкосигнальных тестов, если пользователь явно не запросил именно их:

- spec-like tests, которые проверяют архивные markdown/spec документы на наличие заголовков, секций и формулировок вместо поведения production-кода;
- thin-wiring tests, которые проверяют только факт делегирования wrapper-функции во внутреннюю реализацию или проброс аргументов через monkeypatch без проверки пользовательского эффекта.

Обязательное правило выбора теста:

- если тест может упасть из-за редакторской правки архивной документации, а не из-за регрессии приложения, ему не место в основном regression-suite;
- если тест подтверждает только `function A called function B with same args`, но не страхует observable contract, его не нужно добавлять вместо integration/behavior test;
- приоритет имеют tests на пользовательский success/failure path, runtime events, state transitions, output DOCX/artifacts и реальные guard-контракты.

## 0.1. Защищённый startup performance contract

Startup contract считается отдельной защищённой поверхностью репозитория.

Канонический документ: `docs/STARTUP_PERFORMANCE_CONTRACT.md`.

ИИ-агенту запрещено в рамках несвязанной задачи:

- возвращать тяжёлый синхронный startup path в `app.py`;
- убирать кеширование one-time ресурсов в `app.py`, `config.py` и `generation.py`;
- возвращать Streamlit file watching или `runOnSave` в штатный runtime;
- менять WSL-first runtime contract ради обходного локального удобства.

Если пользователь явно просит изменить startup contract, ИИ-агент обязан синхронно обновить код, startup-related tests и canonical docs, включая `README.md`, `CONTRIBUTING.md`, `docs/WORKFLOW_AND_IMAGE_MODES.md` и `.github/copilot-instructions.md`.

## 0.2. WSL-first project runtime — источник истины для разработки и валидации

Для этого репозитория канонический runtime — project runtime в WSL по пути `/mnt/d/www/projects/2025/DocxAICorrector`.

ИИ-агенту и любому automation/tooling в проекте обязательно:

- при разработке, проверках, тестах, диагностике и live-валидации по умолчанию использовать именно этот WSL-first runtime;
- запускать Python-команды, `pytest`, диагностические импорты и runtime-проверки через проектную `.venv` внутри WSL;
- не делать выводы о зависимостях, import health, доступности Pandoc или общем состоянии окружения на основании случайного `python`, `py`, Windows virtualenv или иного системного интерпретатора, пока не проверен project runtime;
- при расхождении между системным интерпретатором и project runtime считать источником истины project runtime;
- предпочитать существующие project entry points вместо ad-hoc команд: `bash scripts/test.sh ...`, `scripts/start-project.ps1`, `scripts/status-project.ps1`, `scripts/project-control-wsl.sh`.

Короткие корректные примеры:

```bash
bash scripts/test.sh tests/ -q
bash -lc 'cd /mnt/d/www/projects/2025/DocxAICorrector && . .venv/bin/activate && python -m pytest tests/test_config.py -vv'
```

## 1. Базовые инженерные принципы

### 1.1. Не усиливай `app.py` как god-object

ИИ-агент **не должен** добавлять в `app.py` новую бизнес-логику, если её можно разместить в специализированном модуле.

Обязательные правила:

- orchestration UI допустима в `app.py`;
- domain-решения, scoring, policy, validation и pipeline-ветвления нужно выносить в профильные модули;
- если новая функция одновременно знает про UI, OpenAI, document reinsertion и state mutation, это признак неправильного placement.

**Правило принятия решения:**

Если новая логика может быть протестирована без Streamlit UI, она не должна рождаться в `app.py`.

---

### 1.2. Один модуль — одна основная ответственность

Перед добавлением новой функции ИИ-агент обязан ответить:

1. Это orchestration, domain logic, integration, parsing или rendering?
2. В каком модуле эта ответственность уже живёт?
3. Не появляется ли скрытая связка между слоями?

Если ответ неочевиден, нужно:

- предпочесть выделение helper/service слоя;
- не дублировать принятие решений в нескольких местах;
- не смешивать policy и transport-код в одной функции.

---

### 1.3. Не полагайся на неявные downstream-компенсации

Если в pipeline есть строгий validator, soft-accept, fallback и reinsertion, ИИ-агент не должен считать, что один этап автоматически компенсирует дефект другого.

Запрещённая логика:

- `это безопасно, потому что потом всё равно будет fallback`;
- `validator можно ослабить, потому что soft-accept потом решит`;
- `metadata можно не сохранять, потом восстановим по месту`.

Каждый этап обязан быть корректным сам по себе.

---

### 1.4. UI styling должен идти через явный контракт компонента

ИИ-агент не должен решать UI-проблемы спонтанным HTML/CSS-патчем, если задача может быть решена нативными Streamlit-компонентами или одним централизованным component-contract.

Обязательные правила:

- сначала предпочитать native Streamlit (`st.info`, `st.warning`, `st.caption`, `st.metric`, `st.progress`, `st.columns`, `st.expander`) вместо `unsafe_allow_html=True`;
- если поверхность изолирована через `components.html(...)`, считать её отдельным документом с отдельным theme/layout contract;
- для `components.html(...)` держать один централизованный helper/constant для theme shell, а не разбрасывать inline CSS по разным call site;
- не задавать `font-family` в component-level CSS без явной задачи на типографику;
- не полагаться на наследование CSS между основным Streamlit UI и iframe из `components.html(...)`;
- не вводить новые глобальные стили только ради одного локального дефекта, если можно убрать сам HTML-обход и вернуться к native компонентам.

Правило принятия решения:

1. Если native Streamlit закрывает задачу без потери основного UX-контракта, использовать native Streamlit.
2. Если нужен client-side state без server rerun и приходится использовать `components.html(...)`, вводить один централизованный component contract.
3. Только если оба варианта невозможны, допустим локальный CSS/HTML workaround, и он обязан быть явно ограничен одной поверхностью.

## 2. Правила для image/OpenAI pipeline

### 2.1. Явно разделяй direct и structured ветки

ИИ-агент обязан поддерживать раздельную ментальную модель для:

- `semantic_redraw_direct`;
- `semantic_redraw_structured`;
- `safe` path.

Нельзя:

- описывать весь image pipeline как один вызов OpenAI;
- смешивать требования direct и structured веток в одной policy без явной причины;
- добавлять общую логику, которая меняет поведение всех веток без фиксации impact.

Для каждой новой правки должно быть явно указано:

- к какой ветке она относится;
- какой у неё request path;
- какой fallback path;
- как меняется validation path.

---

### 2.2. Любой внешний AI-вызов считается ненадёжным по умолчанию

Для каждого вызова OpenAI ИИ-агент обязан проверить наличие:

- timeout;
- retry policy;
- backoff;
- отдельной обработки transient ошибок;
- отдельной обработки policy/validation ошибок;
- логирования причины fallback.

Это относится ко всем вызовам вида:

- `responses.create`;
- `images.generate`;
- `images.edit`;
- и любым будущим multimodal endpoints.

**Запрещено** добавлять новый внешний вызов по схеме `call -> except Exception -> fallback`, если не разобраны хотя бы основные классы отказов.

---

### 2.3. SDK-compatibility retry не заменяет operational retry

Если код умеет:

- убрать unsupported optional param;
- сократить prompt;
- подобрать fallback size;

это **не считается** полноценной устойчивостью интеграции.

ИИ-агент не должен помечать такую реализацию как `retry-ready` или `resilient`, пока отдельно не покрыты:

- сеть;
- rate limit;
- server-side transient failures;
- hanging requests.

---

### 2.4. Metadata contract для изображений обязателен

Любой код, который создаёт или модифицирует candidate image, обязан явно сохранять и передавать дальше:

- фактический MIME результата;
- исходный размер;
- итоговый размер;
- режим генерации;
- outcome strict validation;
- факт soft-accept;
- источник fallback decision;
- состояние placeholder integrity, если оно на это влияет.

**Запрещено** повторно анализировать candidate, используя предположения вместо фактической metadata.

Отдельное правило:

- если output format может измениться, нельзя повторно использовать старый MIME type из source asset.

---

### 2.5. Не выдавай heuristic analysis за AI analysis

Если функция не вызывает Vision/AI и работает на эвристиках, ИИ-агент обязан:

- явно описывать её как heuristic-only;
- не вводить параметры, создающие ложное впечатление AI-анализа;
- не писать в документации, что semantic equivalence проверена моделью, если это не так.

**Запрещено** использовать нейминг и комментарии, которые завышают реальную надёжность heuristic stage.

---

### 2.6. Validator не является единственной точкой истины

Если pipeline содержит strict validator и soft-accept path, ИИ-агент обязан:

- явно документировать оба механизма;
- не анализировать качество pipeline только по strict validator;
- логировать финальное решение вместе с причиной принятия.

Минимальное требование:

- должно быть видно, был ли candidate принят строго, мягко или через fallback.

---

### 2.7. Стоимость — часть архитектуры, а не вторичный вопрос

Для всех AI-веток ИИ-агент обязан оценивать:

- число внешних вызовов на одну попытку;
- возможные каскадные fallback-цепочки;
- high-fidelity или high-cost режимы;
- необходимость budget guard или cost logging.

**Запрещено** добавлять новую ветку генерации без понимания её worst-case call chain.

## 3. Правила для DOCX и reinsertion

### 3.1. Не разрушай XML-структуру документа при локальной правке

Если ИИ-агент меняет XML узлы DOCX, он обязан различать:

- content nodes;
- formatting nodes;
- structural nodes.

**Запрещено** удалять все дочерние элементы контейнера, если задача состоит только в замене runs/content.

Перед удалением XML-узлов нужно ответить:

1. Какие свойства документа живут в этих узлах?
2. Какие из них нужно сохранить?
3. Есть ли минимально инвазивный способ заменить только содержимое?

---

### 3.2. Reinsertion обязан сохранять layout contract

При повторной вставке изображения ИИ-агент обязан сохранять:

- paragraph properties;
- placement semantics;
- размеры изображения;
- привязку к placeholder;
- ожидаемый порядок элементов в документе.

Если какой-то из этих параметров не сохраняется, изменение считается неполным.

---

### 3.3. Placeholder integrity — это gate, а не просто лог

Если placeholder mismatch делает невозможной корректную сборку документа, ИИ-агент не должен ограничиваться логированием.

Нужно выбирать одно из явных действий:

- abort с понятной ошибкой;
- fallback к безопасному варианту;
- принудительное сохранение исходного asset;
- явная пометка результата как degraded.

**Запрещено** молча продолжать pipeline, если это может привести к пропаже изображения из итогового документа.

---

### 3.4. Любой пользовательский DOCX считается потенциально враждебным входом

Перед добавлением новых операций чтения документа ИИ-агент обязан учитывать:

- ограничение размера входного файла;
- риски распаковки zip-based формата;
- предсказуемое поведение на повреждённых файлах;
- контролируемую ошибку вместо неконтролируемой деградации памяти.

---

## 4. Правила для runtime, state и отказоустойчивости

### 4.1. Worker lifecycle должен быть завершённым даже при аварии

Если ИИ-агент добавляет background worker, thread, queue consumer или async task, он обязан обеспечить:

- гарантированный финальный сигнал завершения;
- cleanup state в `finally` или эквивалентной конструкции;
- логирование crash-path;
- отсутствие зависания UI в состоянии `running` навсегда.

**Запрещено** строить cleanup только на happy-path сигнале.

---

### 4.2. Session state не должен дублироваться вручную в нескольких местах

Если новый state key появляется в инициализации, ИИ-агент обязан проверить:

- где он сбрасывается;
- где сериализуется;
- где участвует в UI;
- не появился ли второй список дефолтов отдельно от первого.

Правильный подход:

- единый источник truth для state defaults;
- массовая инициализация и reset из одного контракта.

---

### 4.3. Логика принятия решения и логика отображения не должны сливаться

Если код влияет на результат обработки документа, он не должен зависеть от UI-specific деталей вроде widget nonce, render key или визуального состояния компонента.

Это особенно важно для:

- progress handling;
- worker completion;
- финального статуса обработки;
- fallback messaging.

---

## 5. Правила для тестов и регрессий

### 5.1. Каждый найденный архитектурный баг должен получать regression test

Если исправляется ошибка из код-ревью, ИИ-агент обязан добавить тест, который фиксирует именно этот класс поломки.

Минимально обязательные типы тестов для этого проекта:

- сохранение paragraph properties после reinsertion;
- сохранение размеров изображения после reinsertion;
- корректность MIME metadata после semantic generation;
- корректный финальный статус при crash-path worker-а;
- корректное различение strict accept, soft-accept и fallback.

---

### 5.2. Тест должен проверять contract, а не внутреннюю случайность реализации

ИИ-агент не должен писать тесты только на количество helper-вызовов или на несущественные промежуточные детали, если бизнес-риск лежит в другом месте.

Правильный вопрос:

- какой пользовательский или архитектурный contract должен остаться истинным после изменения?

---

### 5.3. Для интеграционных веток нужны негативные сценарии

Если есть внешняя интеграция или fallback path, тестовый набор должен включать не только happy-path, но и:

- отказ direct ветки;
- переход в structured ветку;
- потерю placeholder;
- недоступность API;
- повреждённый image payload;
- пустой или несовместимый response.

---

### 5.4. В этом репозитории полный pytest нужно запускать напрямую в WSL через test.sh

Для этого проекта недостаточно абстрактной команды `pytest` из любой оболочки.

Обязательное правило:

- если нужно прогнать весь test suite, ИИ-агент должен использовать `bash scripts/test.sh tests/ -q` из WSL или task `Run Full Pytest`;
- для file-level и node-level сценариев агент должен использовать `bash scripts/test.sh <target> -vv` или соответствующие `Run Current Test File` и `Run Current Test Node` tasks;
- нельзя полагаться на `.venv\Scripts\python.exe` как на источник истины, потому что этот репозиторий использует WSL-based virtualenv;
- перед интерпретацией результатов нужно убедиться, что pytest видит корень проекта в `sys.path` и может импортировать `app.py`, `generation.py`, `restart_store.py` и остальные top-level модули.

Канонический запуск:

```bash
bash scripts/test.sh tests/ -q
```

Низкоуровневый fallback:

```bash
bash -lc 'cd /mnt/d/www/projects/2025/DocxAICorrector && . .venv/bin/activate && pytest tests -q'
```

Если нужен подробный прогресс по каждому тесту, агент должен переключаться на `pytest tests -vv`.

Если запуск делается из Windows shell или внешней automation вне WSL, допустим путь `PowerShell -> WSL -> .venv/bin/activate -> pytest`, но это не основной сценарий для VS Code в этом репозитории.

## 6. Правила для документации и ревью

### 6.1. Не используй ложную точность

ИИ-агент не должен писать в ревью и документации:

- слишком уверенные impact-формулировки без проверки реального blast radius;
- устаревающие line references там, где достаточно file + function;
- описания pipeline, которые уже не соответствуют коду.

Если утверждение не подтверждено полностью, оно должно быть помечено как гипотеза, риск или вероятная проблема, а не как факт.

---

### 6.2. Документируй фактический current state, а не желаемую архитектуру

Если документ описывает текущее состояние системы, ИИ-агент обязан отделять:

- что уже реализовано;
- что частично реализовано;
- что является planned direction.

**Запрещено** смешивать current state и target architecture без явной маркировки.

---

## 7. Анти-паттерны, которые ИИ-агенту запрещено повторять

Нельзя:

1. Добавлять ещё одну policy-ветку прямо в `app.py`, если она может жить в выделенном service/module.
2. Передавать stale metadata downstream после генерации изображения.
3. Удалять XML-узлы абзаца целиком ради замены только визуального содержимого.
4. Считать логирование достаточной реакцией на data-integrity проблему.
5. Путать heuristic checks с AI validation.
6. Объявлять интеграцию устойчивой только потому, что она умеет убирать unsupported params.
7. Добавлять дорогой AI path без budget/logging стратегии.
8. Оставлять crash-path worker-а без гарантированного cleanup.
9. Исправлять критичный баг без regression test.
10. Документировать код с заведомо устаревающими или непроверенными утверждениями.

---

## 8. Чеклист ИИ-агента перед любым изменением

Перед завершением задачи ИИ-агент обязан пройти этот чеклист:

### Архитектура

- Новая логика не усилила `app.py` без необходимости.
- Ответственность новой функции размещена в правильном модуле.
- Policy, transport и UI не смешаны без явного основания.

### Image/OpenAI

- Для новой AI-ветки понятны request path и fallback path.
- Фактические MIME и размеры не теряются после генерации.
- Есть корректное различение heuristic stage и AI stage.
- Учтены timeout, retry, backoff и budget implications.

### DOCX

- Не ломаются paragraph properties и структура документа.
- Reinsertion сохраняет размеры и привязку.
- Placeholder mismatch не игнорируется.

### Runtime

- Worker или async path не зависает при исключении.
- Cleanup и финальный статус гарантированы.

### Тесты

- Добавлен regression test на исправляемый класс ошибки.
- Проверен хотя бы один негативный сценарий.

### Документация

- Описание соответствует фактическому текущему коду.
- Нет ложной точности и неподтверждённых категоричных выводов.

## 9. Короткая инструкция для системного промпта ИИ-агента

Если нужно встроить эти правила в системный или project prompt, используйте краткую форму:

> Не усиливай `app.py` как orchestration-монолит. Не смешивай heuristic stages с AI validation. Для любого OpenAI-вызова учитывай timeout, retry, backoff, fallback и стоимость. Всегда сохраняй metadata contract изображения: MIME, размеры, режим генерации и итоговое решение pipeline. Не ломай DOCX formatting nodes при reinsertion. Любой placeholder mismatch трактуй как decision point, а не просто log. Каждый исправленный архитектурный дефект закрепляй regression test-ом.

---

## 10. Ожидаемый эффект от соблюдения правил

Эти правила должны предотвращать повторение следующих классов ошибок:

- разрастание orchestration-layer и возврат к god-object архитектуре;
- некорректные fallback-допущения в image pipeline;
- потеря MIME/size/placeholder metadata;
- деградация DOCX formatting при reinsertion;
- ложная уверенность в устойчивости OpenAI integration;
- тихие runtime-зависания при авариях worker-а;
- отсутствие регрессионной защиты после исправления критичных дефектов;
- расхождение документации с реальным кодом.
